**Database-Level Authorization with Supabase: RLS and Multi-Tenant Security**

1. **Why DB-Level Authorization Is Mandatory**

Modern SaaS breaches often follow a common pattern: a single missed authorization check in application code leads to one tenant accessing another tenant’s data[ 1 ](https://www.thecloudplaybook.com/p/api-gateway-custom-authorizers-for-tenant-aware-security#:~:text=Last%20month%2C%20while%20reviewing%20our,that%20made%20my%20stomach%20drop). Relying *only* on API-level checks means every endpoint and query must handle tenant isolation correctly – a tall order prone to human error![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.003.png)[ 2 ](https://aws.amazon.com/blogs/database/multi-tenant-data-isolation-with-postgresql-row-level-security/#:~:text=In%20a%20shared%20database%20model%2C,variability%20of%20your%20source%20code). ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.004.png)**Database-level authorization**  (e.g. PostgreSQL Row Level Security) provides a safety net by centrally

enforcing data isolation on every query. This “defense in depth” ensures that even if a developer forgets a filter or an attacker finds a way around the app logic, the database itself will not return unauthorized rows

[3 ](https://dev.to/blackie360/-enforcing-row-level-security-in-supabase-a-deep-dive-into-lockins-multi-tenant-architecture-4hd2#:~:text=While%20application,access%2C%20the%20data%20remains%20protected). In other words, critical multi-tenant rules aren’t merely scattered in code; they live in the database![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.005.png)

engine where they *always* apply.

Real-world incidents underscore the need for this approach. For example, a single flaw in an API Gateway authorizer allowed a Tenant A user to retrieve Tenant B’s records – a catastrophic cross-tenant data leak fixed by adding one missing condition[ 1 ](https://www.thecloudplaybook.com/p/api-gateway-custom-authorizers-for-tenant-aware-security#:~:text=Last%20month%2C%20while%20reviewing%20our,that%20made%20my%20stomach%20drop). RLS could have prevented such a leak by **automatically scoping queries to the tenant context![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.006.png)**. Application-layer checks alone are insufficient because they can be bypassed through coding mistakes, forgotten conditions, or unforeseen request paths. By contrast, when the database requires every query to include the proper tenant constraints, the risk of accidental data exposure drops dramatically[ 4 ](https://aws.amazon.com/blogs/database/multi-tenant-data-isolation-with-postgresql-row-level-security/#:~:text=A%20pooled%20model%20saves%20the,implemented%20in%20every%20SQL%20statement). ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.007.png)

Implementing RLS is admittedly challenging – it shifts complexity into SQL policies and demands careful schema design[ 5 ](https://mergify.com/blog/application-vs-database-where-should-permissions-live#:~:text=In%20short%2C%20RLS%20does%20not,code%20to%20the%C2%A0PostgreSQL%20schema%20design). Yet this pain is necessary for security. It’s far too easy for developers to “get the first![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.008.png)

part right” (authentication) and botch the rest – namely enforcing *which tenant’s data* a user can access ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.009.png)6 . With RLS, that vital tenant context is never an afterthought; it’s baked into the data layer. In summary, **DB- level authorization is mandatory** in a multi-tenant SaaS because it guarantees **tenant isolation at the last line of defense**. It compensates for any lapses in upstream logic, protecting against cross-tenant breaches that would otherwise destroy customer trust[ 7 ](https://www.thecloudplaybook.com/p/api-gateway-custom-authorizers-for-tenant-aware-security#:~:text=Most%20developers%20nail%20the%20first,completely%20botch%20the%20second%20two). While it requires more upfront work and expertise, this approach is considered non-negotiable for robust SaaS security![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.010.png)[ 8 ](https://makerkit.dev/blog/tutorials/supabase-rls-best-practices#:~:text=,your%20database%20to%20untrusted%20clients), especially in high-stakes domains like finance and ERP where mistakes can be costly.![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.011.png)

2. **Supabase Auth Architecture**

Supabase provides an out-of-the-box architecture that marries authentication with database enforcement. It consists of several layers: a **client SDK**, a secure auth service, and Postgres itself. The **Supabase Auth service** (a fork of Netlify’s GoTrue) handles identity: it validates credentials (email/password, OAuth logins, etc.), communicates with external providers (Google, GitHub, Apple, SAML SSO, etc.), and issues JWT access tokens[ 9](https://supabase.com/docs/guides/auth/architecture#:~:text=The%20Auth%20service%20is%20an,project%2C%20originally%20created%20by%20Netlify)[ 10 ](https://blog.logrocket.com/exploring-supabase-the-open-source-firebase-alternative/#:~:text=Supabase%20provides%20a%20simple%20API,supports%20enterprise%20logins%20like%20SAML). You can use the hosted Supabase cloud or self-host the stack – either way, the auth service runs alongside your Postgres database. In a hosted Supabase project, all components (Postgres, auth, REST API) are managed for you, whereas self-hosting means you run Docker containers or servers for the auth service and PostgREST. Self-hosting grants more control (useful for on-premises deployments or strict![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.012.png) compliance needs) at the cost of more maintenance. Either mode supports a wide range of sign-in options (from social logins to magic links), which Supabase standardizes into its JWT-based security model.

**How does a Supabase JWT translate into DB permissions?** The process is designed so that Postgres itself can validate and utilize the token. When a user logs in (or signs up) via Supabase Auth, the service returns a signed ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.013.png)**JWT** (JSON Web Token) containing the user’s unique ID ( sub claim) and other attributes like their role or any custom claims. The JWT is signed with a secret that the Postgres database knows (Supabase configures the database with this secret). When the client (frontend or backend) makes requests to the database (through Supabase’s auto-generated REST endpoints or via a Postgres client), it supplies the JWT. The Supabase API gateway (or the custom backend server) will **map that JWT into the database session** by setting  a  special  PostgreSQL  config  with  the  token’s  claims.  In  fact,  Supabase’s  integration  uses![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.014.png)

current\_setting('request.jwt.claims') in Postgres to expose the token payload to SQL [11](https://medium.com/@gracew/using-supabase-rls-with-a-custom-auth-provider-b31564172d5d#:~:text=We%20need%20to%20use%20this,use%20later%20in%20step%204)[ 12 ](https://supabase.com/docs/guides/realtime/authorization#:~:text=JWT%20claims,claims%20setting). This means that inside any SQL policy or function, we can call functions like ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.015.png)![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.016.png) auth.uid() to get the user’s ID from the JWT and  auth.role() or  auth.jwt() to get other claims, directly within the database [13](https://supabase.com/docs/guides/database/postgres/row-level-security#:~:text=create%20policy%20,their%20own%20todos)

[14 ](https://supabase.com/docs/guides/auth/oauth-server/token-security#:~:text=,is%20in%20the%20token). The database trusts the JWT because it’s signed by the known secret – so if the token was tampered with, signature verification would fail.

**Hosted vs. Self-Hosted:** In a hosted Supabase environment, the JWT signing key is managed for you and rotated by Supabase as needed. In self-hosted mode, you obtain or set the JWT secret yourself (Supabase’s docs show where to find it) [15 ](https://medium.com/@gracew/using-supabase-rls-with-a-custom-auth-provider-b31564172d5d#:~:text=1,Secret). Either way, the trust relationship is the same: Supabase Auth is the issuer of tokens, and Postgres is the consumer that enforces them. One benefit of this tight integration is that ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.017.png)**trust boundaries** are clearly defined – the client and any external network are untrusted, but the token itself carries cryptographic proof of identity and role. The database doesn’t need to trust the application code; it only trusts the token it receives (and the token’s issuer). This mitigates whole classes of attacks. For instance, even a compromised or buggy backend cannot bypass RLS without a valid token or the highly- privileged service key.

![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.018.png)

These roles exist in the database and are used in policy definitions (for example, you can write ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.021.png) TO authenticated to apply a policy only to logged-

in users). The JWT’s “role” claim usually is set to “authenticated” for normal users – the Supabase client library will use the anon key for public requests and switch to the user’s token for authenticated requests, automatically assuming the proper role on the Postgres side [17 ](https://supabase.com/docs/guides/database/postgres/row-level-security#:~:text=,anon%20key). Supabase’s auth schema (in the![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.022.png)![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.023.png)

auth.users table) stores the user accounts, but that schema is not exposed to the public API directly [18 ](https://supabase.com/docs/guides/auth/architecture#:~:text=Postgres). Instead,![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.024.png)![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.025.png)  you  join  or  reference  auth.users  from  your  tables  as  needed  (for  example,  a

profiles.user\_id  foreign key to  auth.users ). This way, you can connect business data to the identity system, but the low-level user data (emails, hashes, etc.) remain protected behind RLS (or additional privileges) unless you explicitly expose them.

In summary, Supabase’s auth architecture provides a **secure identity layer tightly coupled with Postgres**. The JWT contains all information needed to enforce permissions in SQL, eliminating the need for custom middleware  in  many  cases.  Hosted  Supabase  offers  convenience  and  managed  security  (including continuous updates and compliance certifications on their end), whereas self-hosting gives you full control over deployment (you might choose this for on-prem compliance like PCI-DSS if you need to manage all data internally). Either approach, combined with Postgres RLS, lets you achieve a robust, **end-to-end authorization pipeline** from the UI all the way down to each table row.

3. **PostgreSQL RLS Deep Dive**

PostgreSQL’s Row Level Security (RLS) is the cornerstone of our defense-in-depth strategy. It allows defining ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.026.png)![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.027.png)**policy rules per table**  that filter which rows a given database role (or user) can  SELECT ,  INSERT , ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.028.png)![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.029.png)

UPDATE, or  DELETE . In a multi-tenant SaaS, we use RLS to enforce **tenant isolation** at the database level, as well as finer-grained rules like user ownership and role-based permissions within a tenant.

**Tenant Isolation Policies:** Every table that contains tenant-specific data gets an RLS policy ensuring only rows belonging to the user’s own tenant are accessible. Typically, this means each table has a ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.030.png) tenant\_id column. The RLS policy checks that the ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.031.png) tenant\_id on the row matches the tenant context of the current

user. How do we get the user’s tenant context? There are a couple of patterns. One approach is to include the tenant ID as a claim in the JWT (if your auth system populates it), then use ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.032.png) auth.jwt() in the policy. For example, if the JWT has  "tenant\_id":"..." inside, a policy might be: ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.033.png)

CREATE POLICY "Tenant isolation on Orders"![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.034.png)

ON orders FOR SELECT TO authenticated

USING ( (auth.jwt() ->> 'tenant\_id')::uuid = orders.tenant\_id );

This policy (attached to the  orders  table) transparently adds a  WHERE orders.tenant\_id = ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.035.png)![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.036.png)![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.037.png)'<user\_tenant>' condition to every SELECT for authenticated users. If the user’s token says they belong to tenant  ABC, Postgres will only return orders with  tenant\_id = ABC [19](https://aws.amazon.com/blogs/database/multi-tenant-data-isolation-with-postgresql-row-level-security/#:~:text=,tenant%20ENABLE%20ROW%20LEVEL%20SECURITY)[ 20 ](https://makerkit.dev/blog/tutorials/supabase-rls-best-practices#:~:text=create%20policy%20,documents). If a user somehow tries to query another tenant’s ID, the condition fails and they get an empty result. **No application bug can circumvent this** because the database itself refuses to return non-matching rows.

In cases where the JWT does *not* directly carry a tenant ID, you can derive it through a join or a helper function. For instance, you might have a join table ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.038.png) membership(user\_id, tenant\_id, role) that records which tenant(s) a user belongs to. You can then write a policy like *“allow select if the row’s tenant\_id is*

*in the set of tenant\_ids for which this user is a member.”* Using a subquery or a custom SQL function can achieve this. For example, a policy on ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.039.png) projects could use a sub-select on the membership table:

CREATE POLICY "Org members can view project" ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.040.png)ON project FOR SELECT TO authenticated

USING (

project.tenant\_id IN (

SELECT m.tenant\_id FROMmembership m WHEREm.tenant\_id = project.tenant\_id

AND m.user\_id = auth.uid()

)

);

This ensures the user making the request ( auth.uid() provides their user UUID from the JWT) has a membership in the same tenant as the project![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.041.png) [21 ](https://dev.to/blackie360/-enforcing-row-level-security-in-supabase-a-deep-dive-into-lockins-multi-tenant-architecture-4hd2#:~:text=CREATE%20POLICY%20,org_id%20IN). If not, that project is invisible to them. The example above illustrates checking tenant isolation via a join, which is powerful but can impact performance if not indexed; we’ll address performance shortly.

**User-Owned Rows:** Within a tenant, there are often rows that belong to a specific user (e.g. a user’s own profile or personal settings). RLS can enforce that *users can only see or modify their own records*. This is usually a simple policy comparing a user identifier. For example, a ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.042.png)![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.043.png) profiles table might have  user\_id as a foreign key to ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.044.png) auth.users . The policy for user-owned access would be: 

CREATE POLICY "Only owner can view profile"![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.045.png)

ON profiles FOR SELECT TO authenticated USING ( auth.uid() = profiles.user\_id );

This uses the Supabase-provided  auth.uid()  function, which returns the current JWT’s subject (user UUID)![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.046.png)[ 13 ](https://supabase.com/docs/guides/database/postgres/row-level-security#:~:text=create%20policy%20,their%20own%20todos). The policy effectively does  ... WHERE profiles.user\_id = current\_user\_id . If you attempt to select someone else’s profile, the condition fails and Postgres returns no rows. Similar policies![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.047.png)![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.048.png)

can be set for updates and deletes (often using  WITH CHECK for new data). For instance, to allow updates to one’s own profile: 

CREATE POLICY "Only owner can update profile"![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.049.png)

ON profiles FOR UPDATE TO authenticated USING ( auth.uid() = profiles.user\_id )

WITH CHECK ( auth.uid() = profiles.user\_id );

The  USING clause ensures the row being fetched belongs to the user, and the  WITH CHECK ensures that any ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.050.png)![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.051.png)*new values* being written still satisfy the condition (so you can’t transfer a profile to someone else’s user\_id by updating it) [22](https://supabase.com/docs/guides/database/postgres/row-level-security#:~:text=Let%27s%20say%20you%20have%20a,to%20update%20their%20own%20profile)[ 23 ](https://supabase.com/docs/guides/database/postgres/row-level-security#:~:text=10). ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.052.png)

**Role-Based Row Access:**  Beyond simple ownership, enterprise apps often have roles like “Admin”, “Manager”, “Viewer” with different access rights. We can encode these in RLS either via additional JWT claims or by lookups. One strategy is to add custom JWT claims for user roles or permissions (Supabase allows adding custom claims on signup or via database triggers) [24 ](https://github.com/orgs/supabase/discussions/1148#:~:text=Custom%20claims%20for%20multi%20tenancy,Describe%20the%20solution%20you%27d). For example, a JWT could include![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.053.png)![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.054.png)![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.055.png)

"role": "admin" or even an array of tenant role memberships. Then a policy can check  auth.jwt() - >> 'role' = 'admin'  to grant broader access. Supabase’s documentation shows using claims to differentiate access by OAuth clients as well [25 26 ](https://supabase.com/docs/guides/auth/oauth-server/token-security#:~:text=auth), which is analogous – you might have a claim that indicates the context or privileges of the token.![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.056.png)

However, storing roles in JWTs that can change (like if you update a user to be an admin) means you’d have to reissue the token. An alternative is to use security definer functions in the policy that check the database membership  table.  The **LockIn  example** 27[ 28 ](https://dev.to/blackie360/-enforcing-row-level-security-in-supabase-a-deep-dive-into-lockins-multi-tenant-architecture-4hd2#:~:text=Before%20implementing%20policies%2C%20we%20created,functions%20to%20check%20user%20permissions)[ d](https://dev.to/blackie360/-enforcing-row-level-security-in-supabase-a-deep-dive-into-lockins-multi-tenant-architecture-4hd2#:~:text=,user_id)emonstrates  this:  they  created  functions![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.057.png)![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.058.png)

is\_organization\_member(org\_id, user\_id) ,  is\_organization\_admin(org\_id, user\_id) etc., which query the membership table to see if the current user has the required role. Those functions are then used in RLS policies. For instance, a policy allowing admins to update a project could look like:

CREATE POLICY "Admins can update project"![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.059.png)

ON project FOR UPDATE TO authenticated

USING ( public.is\_organization\_admin(project.tenant\_id, auth.uid()) )

WITH CHECK ( public.is\_organization\_admin(project.tenant\_id, auth.uid()) );

This uses a custom function  is\_organization\_admin(org\_id, uid) that returns true only if that user is an admin of the given organization![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.060.png) [29](https://dev.to/blackie360/-enforcing-row-level-security-in-supabase-a-deep-dive-into-lockins-multi-tenant-architecture-4hd2#:~:text=,member%20WHERE%20organization_id%20%3D%20org_id)[ 30 ](https://dev.to/blackie360/-enforcing-row-level-security-in-supabase-a-deep-dive-into-lockins-multi-tenant-architecture-4hd2#:~:text=CREATE%20POLICY%20,is_organization_admin%28org_id). Both the USING and WITH CHECK enforce that rule (so they can only update if they are admin, and they cannot change the project’s tenant\_id or other fields in a way

that violates the rule either). We could likewise have a more permissive policy for regular members:

CREATE POLICY "Members can view project"![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.061.png)

ON project FOR SELECT TO authenticated

USING ( public.is\_organization\_member(project.tenant\_id, auth.uid()) );

By layering multiple policies on the same table, each for different roles/actions, we achieve role-based access control (RBAC) entirely in the database. The advantage is that **any query route (direct or through any microservice)**  will still enforce these rules. Even someone using a psql console with the user’s credentials would be restricted.

**Service Account Policies:** In addition to human users, systems often have background services (cron jobs, integrations, etc.) that need broader access. Supabase addresses this with a special *service role*. The service role (associated with the “service\_role” API key) is a Postgres role that bypasses RLS entirely [31 ](https://makerkit.dev/blog/tutorials/supabase-rls-best-practices#:~:text=The%20rule%20is%20simple%3A%20enable,bypasses%20RLS%20for%20admin%20operations)– it’s meant for trusted server-side operations like running bulk migrations or admin tasks. In our architecture, we ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.062.png)**never expose the service role key to the public**; it’s kept in server-side code or cloud functions. When used, any query with the service role will ignore RLS policies, effectively allowing full access. This is powerful

but dangerous if misused, so it should be limited to specific internal use-cases. For example, an internal billing service might use the service role to aggregate financial data across tenants (something normal users can’t do). We’d guard this by network isolation and by not mixing service-role queries with user queries.

Alternatively, you can create *limited* service accounts that still obey RLS but have their own policies. One approach is to create additional Postgres roles, say![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.063.png)![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.064.png)  worker\_role , and issue JWTs with  "role": ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.065.png)"worker\_role" for certain server processes. Then you can write policies  TO worker\_role on tables that grant it broader read access than a normal user, but perhaps still not everything. For instance, you![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.066.png)

might allow  worker\_role to read multiple tenants’ data for an analytic job, but *only specific columns* (you could combine RLS with Postgres **column-level security**  or selective grants in such cases). Designing service account access often means balancing convenience (sometimes it’s easier to just bypass RLS) with

the principle of least privilege. For PCI-DSS compliance, for example, you would want even internal services to have access only to the minimum data necessary – which might mean writing special RLS policies or using separate schemas for highly sensitive info.

**Concrete SQL Example – Putting it Together:** Let’s illustrate a realistic combination of policies on a single table. Consider an ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.067.png)![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.068.png)![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.069.png) Invoices table that has  tenant\_id ,  customer\_id ,  amount , etc. We want: (a) normal users can  SELECT only invoices from their tenant, (b) within a tenant, only users with role “finance”

or “admin” can see invoice details, (c) an admin can  UPDATE invoices (perhaps to correct data) but others cannot. We might achieve this with policies:

ALTER TABLE invoices ENABLE ROWLEVEL SECURITY;![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.070.png)

-- Base tenant isolation for selects (any authenticated user of tenant can see basic info)

CREATE POLICY "Tenant iso for invoices"

ON invoices FOR SELECT TO authenticated

USING ( invoices.tenant\_id = (auth.jwt() ->> 'tenant\_id')::uuid );

-- Finance role can see details (perhaps a separate policy or using column-level SELECT privileges)

-- Admins can update invoices

CREATE POLICY "Tenant admins can update invoices"

ON invoices FOR UPDATE TO authenticated

USING (

invoices.tenant\_id = (auth.jwt() ->> 'tenant\_id')::uuid AND (auth.jwt() ->> 'role') = 'admin'

)

WITH CHECK (

invoices.tenant\_id = (auth.jwt() ->> 'tenant\_id')::uuid );

Here we assumed the JWT has a  'role' claim for simplicity. A more normalized approach would be to join to a roles table as shown earlier. But the key point is that multiple conditions can be combined. The first![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.071.png) policy restricts read access by tenant. The second (update policy) further requires the user’s role be admin – this is an example of enforcing an *application-level rule (only admins can edit)* at the DB level too. If a non- admin somehow calls an update endpoint, the database will reject the update because the RLS ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.072.png) USING expression returns false for them. The![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.073.png)  WITH CHECK  here ensures they can’t maliciously change the tenant\_id of an invoice to another value (though our  USING already prevents selecting cross-tenant, it’s a good practice to mirror tenant check in with-check on updates). 

Finally, note that **RLS policies are additive** by default (OR logic for multiple policies unless marked  AS RESTRICTIVE![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.074.png) ). If a table has multiple policies for the same action, a row is accessible if *any one* policy’s USING clause passes. In complex scenarios, Postgres 15+ allows marking policies as ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.075.png) AS RESTRICTIVE to change combination logic to AND (all must pass) [32](https://supabase.com/docs/guides/auth/oauth-server/token-security#:~:text=Problem%3A%20OAuth%20client%20has%20access,to%20data%20it%20shouldn%27t)[ 33 ](https://supabase.com/docs/guides/auth/oauth-server/token-security#:~:text=CREATE%20POLICY%20). This can be useful to enforce additional global constraints (e.g., a policy that ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.076.png)*no one* except a service can see rows with  is\_archived=true , marked restrictive, can globally filter those out on top of other policies).

**Performance Considerations:** Writing RLS policies for tenant and role checks means additional overhead on every query – but there are best practices to keep this overhead small. We should ensure all columns used in policy conditions are indexed (e.g., index ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.077.png)![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.078.png) tenant\_id , index  user\_id on tables)[ 34 ](https://makerkit.dev/blog/tutorials/supabase-rls-best-practices#:~:text=1,Policies). Supabase recommends wrapping functions like  auth.uid() in a subquery ( (SELECT auth.uid()) ) so that it’s evaluated  once  per  query  instead  of  per  row [35 ](https://supabase.com/docs/guides/database/postgres/row-level-security#:~:text=This%20method%20works%20well%20for,initPlan).  The  snippet  above  used  (auth.jwt()  ->> ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.079.png)'tenant\_id')::uuid which  is  fine;  if  we  call auth.uid() repeatedly,  we’d  do

(SELECT auth.uid()) . Also consider using **security definer** functions for complex permission logic to avoid expensive subqueries in-line – as the LockIn team discovered, moving logic into a function and indexing the relevant keys improved performance significantly [36](https://dev.to/blackie360/-enforcing-row-level-security-in-supabase-a-deep-dive-into-lockins-multi-tenant-architecture-4hd2#:~:text=match%20at%20L537%20CREATE%20POLICY,project_id%20IN)[ 37 ](https://dev.to/blackie360/-enforcing-row-level-security-in-supabase-a-deep-dive-into-lockins-multi-tenant-architecture-4hd2#:~:text=CREATE%20POLICY%20,EXISTS). Always test your policies with realistic data volumes (Postgres’s ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.080.png)![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.081.png) EXPLAIN can show if the policy condition is causing a sequential scan). With careful design, RLS checks can be made very fast and will not be the bottleneck in most systems. And the slight performance cost is usually acceptable given the huge security benefits – RLS essentially *trades a bit of CPU for a lot of safety*. In an ERP-grade system dealing with financial data, that trade-off is well worth it.

4. **ERD for Secure Multi-Tenant SaaS**

![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.082.png)

*Entity-relationship diagram for a secure multi-tenant SaaS model. Each Tenant (organization/account) can have many Users associated via a Membership relationship (a join table recording which users belong to which tenant and with what role). Roles define the permissions level of a membership (e.g. “Owner”, “Admin”, “User”), and a separate Role–Permission mapping can link roles to fine-grained Permissions (CRUD actions, module access, etc.). Application data entities, such as a Document, tie back to the Tenant and optionally to a specific owner User. This structure ensures that every data row is labeled with its owning tenant (and often an owning user), enabling RLS*

*policies to enforce tenant isolation and user-specific access. For example, a document’s*  tenant\_id  *and![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.083.png)![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.084.png)![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.085.png)*

owner\_id *fields allow the database to allow access only if*  document.tenant\_id *matches the user’s tenant* and *the user has the appropriate role or is the owner.*

In the ERD above, we outline the **core tables for multi-tenant security**. At a high level:

- **Tenant** – represents a customer or account in the system. Every piece of primary data is linked to a tenant.
- **User** – represents an end-user/principal (could be a human user or a service principal account). Users may belong to one or multiple tenants.
- **Membership** – a join table linking Users to Tenants, usually with an assigned **Role**. For example, User A might be an “Admin” in Tenant X and also a “Member” in Tenant Y. The Membership table typically has fields: ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.086.png)![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.087.png)![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.088.png) user\_id ,  tenant\_id and  role\_id (or role name). This is crucial for authorization because it defines *which tenant(s) a user can act in, and at what privilege level*.
- **Role** – defines a set of permissions (e.g. Admin, Manager, ReadOnly). Roles can be global or tenant- specific. Often, we define roles globally (same set of roles available for each tenant) and link them via membership. In some systems, tenants can have custom roles, but that adds complexity – our model assumes standard role names are reused across tenants for consistency (which is common in enterprise SaaS).
- **Permission** – (optional in the diagram) represents granular permissions or actions (e.g. “VIEW\_INVOICES”, “EDIT\_USER”, “DELETE\_PROJECT”). Roles can aggregate multiple permissions. We show a RolePermission relationship indicating a many-to-many between roles and permissions. In a simple RBAC, you might not need a separate permissions table (roles could be hard-coded to app logic or just implied), but for an ERP-grade system, it’s often beneficial to have explicit permissions for auditability. This also helps if you need to enforce certain permissions in the database using RLS (for example, a column or row might be accessible only if the user’s role has a certain permission).
- **Document (example data entity)** – this represents any multi-tenant data table (could be Invoices, Orders, etc. – we use “Document” generically). It has a ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.089.png) tenant\_id foreign key (linking to Tenant) and possibly an ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.090.png) owner\_id linking to the User who created or is responsible for the document. The tenant\_id![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.091.png) is the primary tag for data isolation. The optional  owner\_id allows additional rules (like only the owner or an admin can view/edit it).

This schema sets the stage for implementing RLS. For instance, to enforce tenant isolation on the Document table, we ensure every Document row has the correct ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.092.png) tenant\_id . A policy can then simply check that  tenant\_id = auth.jwt()->>'tenant\_id' (or that the user has membership in that tenant

via a join) as discussed. Ownership can be enforced by checking  owner\_id = auth.uid() for certain actions, possibly combined with role checks (e.g.  owner\_id = auth.uid() OR user has admin role 

in that tenant ). The **access paths** in the diagram refer to these routes by which a user gets permission to a resource: either through being the owner, or through being a member of the tenant with the requisite role, or through a specific permission granted to their role.

Another point to note in an ERP/financial context is **principals vs. services**. In our model, “User” can encompass both human users and service accounts. One could differentiate them by a column (e.g.![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.093.png)

type = 'human' or 'service' ). Service principals might not be tied to a single tenant – or they might be tied to a special tenant representing the application itself. Depending on use case, you might have a separate table for service accounts with different relationship rules. But from a high-level ERD perspective, they’re similar: they have roles and possibly tenant associations, and we enforce what they can access in a similar way (with RLS policies or by only giving them certain roles).

Crucially, this ERD supports **least privilege**. By assigning roles to memberships, a user in Tenant X might be an Admin (full access to that tenant’s data) while the same user in Tenant Y is a Viewer (read-only, perhaps enforced by further policies that restrict write operations for “Viewer” role). The RolePermission mapping could even be used to drive dynamic RLS decisions (though typically, RLS policies check role names rather than individual permissions due to the complexity of parsing arrays of permissions in SQL). Still, if you needed to, you could include an array of permissions as a JWT claim and check those in a policy.

Finally, having a concrete data model like this makes it easier to reason about authorization. It becomes straightforward to write queries like “find all users who have the admin role in any tenant” or to ensure via constraints that a user cannot have two memberships with conflicting roles in the same tenant, etc. These consistency checks in the model further reduce the chance of a security mistake. We would also ensure that the membership and role tables themselves are protected by RLS – for example, a user should only be able to see or modify membership records of their own tenant, and only if they are allowed (an ordinary member shouldn’t grant roles to themselves, etc.). This way, even the tables that define permissions are subject to *higher-level* authorization rules, often implemented with careful RLS policies (or sometimes just not exposed via the API at all, with changes only allowed through secure server-side functions).

5. **Application vs. Database Responsibility Split**

A defense-in-depth design means some authorization logic lives in the application and some in the database – with intentional overlap for critical checks. Here’s how to split responsibilities:

- **What** MUST  **be enforced in the database:**  Anything that is fundamental to data security and isolation should be enforced at the database level. This includes tenant isolation (no row from tenant A should ever be visible to tenant B’s user) and object ownership rules (e.g. only the creator of a resource or a role with elevated rights can modify it). By enforcing these rules with RLS, you ensure they’re **always in effect**, no matter how the data is accessed 38[ 8 ](https://makerkit.dev/blog/tutorials/supabase-rls-best-practices#:~:text=RLS%20adds%20implicit%20WHERE%20clauses,This%20means)[. Fo](https://makerkit.dev/blog/tutorials/supabase-rls-best-practices#:~:text=,your%20database%20to%20untrusted%20clients)r example, even if a developer writing a new API endpoint forgets to filter by tenant, the data won’t leak because the DB will return zero rows that don’t match the user’s tenant policy. The database should also enforce referential integrity and business constraints that could be abused – for instance, foreign keys to ensure a user can’t insert an order for another tenant’s customer, or a CHECK that a regular user cannot set themselves as an admin in the membership table (that might be done via a separate controlled process). Essentially, ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.094.png)**any security rule that can be encoded in the database should be**. This includes duplication of certain app logic: if the app says “only admins can delete a project,” the DB should also have a policy or permission check for deletes that ensures the user is admin for that tenant. RLS shines at preventing the “row leakage” or unauthorized data access vulnerability class [39 ](https://mergify.com/blog/application-vs-database-where-should-permissions-live#:~:text=RLS%20should%20not%20be%20treated,leakage%20caused%20by%20missing%20filters), so those protections belong squarely in the DB. Additionally, database-level auditing (like using the ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.095.png) pgaudit extension or logging) should be used to record access to sensitive operations – this is crucial for compliance (PCI-DSS, SOC2) and can only reliably be done at the DB level to capture all access paths.
- **What** MAY  **live in application logic:**  Not all authorization needs to burden the database with complexity. The application can handle higher-level or UI-level authorization that is not just about data visibility. For instance, **feature toggles or UI element hiding** based on role can be done in the

frontend or backend application. If a feature is turned off for a certain role or plan (say, only Gold- tier tenants can access the analytics dashboard), the app can check that and simply not even route the user there. These checks don’t necessarily need an RLS policy (though one could add a safety net at the DB for it too). Another example: **complex workflow rules** – e.g., “a user can only approve an invoice if they haven’t created it themselves” might be easier enforced in application logic (or via a stored procedure with logic) rather than a raw RLS policy. *Business logic heavy conditions*, especially those that span multiple steps or time (like “can only perform action X if 3 other approvals from different users have happened”), typically live in the application service layer. Also, performing calculations or aggregations that involve many rows might be done in the app with appropriate queries – as long as RLS ensures the app only fetched allowed data in the first place, the app can then apply additional rules on that data. In short, **application logic handles the things that are procedural, stateful, or outside the scope of single-row data access**.

- **What must be duplicated (defense in depth):** For critical permissions, *both* the app and DB should enforce them. A prime example is role-based privileges for dangerous actions. The app should not show a “Delete Tenant” button to a non-admin user and should also verify on the API endpoint that the requester is an admin. But we also ensure the database itself will reject the ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.096.png) DELETE on the tenants![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.097.png) table unless the request came from an admin (this could be done by only allowing deletes when using the service role, or an RLS policy that checks a claim) [40 ](https://mergify.com/blog/application-vs-database-where-should-permissions-live#:~:text=%2A%20Access,only%20owners%20can%20delete%20it). This duplication means an attacker would need to exploit both layers to succeed – a bug in the app alone wouldn’t suffice because the DB would stop it, and vice versa.![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.098.png) **Unauthorized row access is one class that absolutely should be double-checked**: the app should request data with proper tenant filters, and the DB will also apply tenant filters. If the app passes an ID that doesn’t belong to the user’s tenant (maybe due to an IDOR bug), the DB’s RLS still blocks it. Another area for duplication is **input validation for security-sensitive fields**. For example, if there’s a field that an ordinary user should never change (like their account’s ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.099.png)![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.100.png) plan\_level or a transaction’s  amount after payment), the app should grey it out or ignore it, *and* the DB should have a CHECK or RLS  WITH CHECK preventing changes to that field except by an appropriate role. Defense in depth often requires repeating ourselves deliberately on these critical points. Yes, it adds some overhead (and needs the dev team to remember to update rules in two places if something changes), but it’s worthwhile for anything that would lead to a privilege escalation or data leak if mis-handled [39 ](https://mergify.com/blog/application-vs-database-where-should-permissions-live#:~:text=RLS%20should%20not%20be%20treated,leakage%20caused%20by%20missing%20filters).![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.101.png)

To put it succinctly: use the database as the **ultimate gatekeeper** – it should be in charge of fundamental data access control. Use the application as the **friendly doorman** – it guides users, provides a better UX by hiding or disabling what they shouldn’t do, and performs pre-checks to fail fast when possible. But if the doorman slips up, the gatekeeper is still there. This layered approach is how you achieve ERP-grade correctness and security. It’s also important for compliance: many regulations require multiple independent controls. For example, PCI-DSS might require both application-level access control *and*  database-level restrictions, plus auditing. Showing that you enforce rules in multiple layers and that no single failure opens a hole is a strong compliance posture.

One must also consider **transparency and maintainability**. Some simple rules (like “user can edit their own profile”) are easy to enforce in both places, which is fine. More complex rules might be defined in one place and the other layer just has to trust it – in such cases, typically the DB rule is the source of truth and the app is just providing user-friendly messaging around it. Documentation and testing become vital so that developers understand which rules live where. But overall, err on the side of putting critical checks in the database. As one analysis put it, application filtering is prone to being accidentally omitted, whereas RLS is

guaranteed by the database every time [41](https://mergify.com/blog/application-vs-database-where-should-permissions-live#:~:text=Safety)[ 42 ](https://mergify.com/blog/application-vs-database-where-should-permissions-live#:~:text=%E2%9A%A0%EF%B8%8F%20Prone%20to%20omission). The downside is that RLS is implicit (developers must remember it’s happening) and can be harder to debug, so a combination of explicit app checks plus implicit DB enforcement works well in practice![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.076.png) [43](https://mergify.com/blog/application-vs-database-where-should-permissions-live#:~:text=%E2%9C%85%20Explicit%20in%20application%20code)[ 44 ](https://mergify.com/blog/application-vs-database-where-should-permissions-live#:~:text=Debugging).![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.102.png)

6. **Threat Modeling & Failure Analysis**

Even with robust architecture, we must anticipate possible failure modes and attack vectors. Below we analyze key threats and how our design mitigates them:

- **Stolen or Leaked Tokens (JWT/API Key):** If a user’s JWT is stolen (say via XSS or an intercepted request), the attacker can impersonate that user – but crucially, they *cannot access other tenants’ data*. The token’s claims (user ID, tenant ID/role) are constrained to that user. RLS ensures the attacker sees only what that user could see [38 ](https://makerkit.dev/blog/tutorials/supabase-rls-best-practices#:~:text=RLS%20adds%20implicit%20WHERE%20clauses,This%20means). This limits damage compared to systems without DB-level isolation (where a stolen token might grant broad database querying). Mitigation: ensure JWTs are short-lived and refreshable so stolen tokens expire quickly, and use secure storage (http-only cookies or keychain) to make theft harder. If a ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.103.png)**service\_role key** (the secret API key that bypasses RLS) were leaked, that’s more severe – it’s essentially a master key. An attacker with the service key could query any table without policies. To mitigate this, we treat service keys like database root passwords: store them only on secure servers, rotate them if there’s any suspicion of leak, and **never expose them to the client**. Supabase’s guidelines explicitly warn that the anon public key is safe to use in client code, but the service\_role key must remain private[ 8 ](https://makerkit.dev/blog/tutorials/supabase-rls-best-practices#:~:text=,your%20database%20to%20untrusted%20clients). Furthermore, we can issue separate service-role keys for different services with limited database roles (Supabase supports multiple keys with different roles). That way, even if one is compromised, it doesn’t give full admin access – e.g., a background job might have a key that only allows read-only service role, not writes. In a PCI context, token leakage is also mitigated by requiring multi-factor auth and potentially IP restrictions for admin users, reducing the chance an attacker can do much with a stolen token. Detection: implement monitoring to flag if a single token is suddenly accessing an unusual amount of data or hitting many tenant IDs – that could indicate misuse.![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.104.png)
- **Broken Access Control in Application Code:** This is the classic case of a developer forgetting to check authorization in an API endpoint (e.g., an endpoint that fetches an order by ID without verifying the order belongs to the current user’s tenant). In our architecture, such mistakes should *not* lead to data leaks. Even if the app passes an ID from another tenant, the database RLS will silently filter out the row (no matching row will be returned) [45](https://supabase.com/docs/guides/database/postgres/row-level-security#:~:text=When%20a%20request%20is%20made,null)[ 46 ](https://supabase.com/docs/guides/database/postgres/row-level-security#:~:text=1). The user either gets an empty result or a not-found error. This dramatically reduces the impact of application bugs. Many high- profile breaches are due to “IDOR” (Insecure Direct Object Reference), where simply incrementing an ID in a URL gives an attacker someone else’s data. With RLS, an IDOR attempt is foiled at the query level – the query for that resource returns nothing because the policy ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.105.png)![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.106.png) WHERE tenant\_id = current\_tenant![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.107.png) fails. That said, it’s important the application *always uses the user’s context when querying*. If a developer accidentally uses the service role or disables RLS for a certain query, then all bets are off. So our mitigation is twofold: (1) Only use restricted roles or user-scoped queries in the app (e.g., if using PostgREST, always use the user’s JWT; if using an ORM/SQL, ensure you execute SET LOCAL request.jwt.claims![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.108.png) or use a SECURITY INVOKER function that applies the needed checks). And (2) we add automated tests for endpoints to ensure they don’t inadvertently bypass auth. Many ORMs don’t natively integrate with RLS, so developers must be cautious – for example, with Prisma or other ORMs, you may need a middleware to set the ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.109.png) request.jwt.claims on each connection [47 ](https://github.com/orgs/supabase/discussions/1148#:~:text=Custom%20claims%20for%20multi%20tenancy,exp%3A%20number%3B%20sub%3A%20string). We’d implement such measures so that application queries run with the user’s role.![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.110.png)

If someone *did* accidentally run a query with the service role in an API, that would be caught in code review or testing (that’s essentially an explicit override, which should be rare). Defense in depth here is basically: rely on RLS, but still perform app-level checks as if RLS wasn’t there, to catch mistakes early and provide helpful errors.

- **SQL Injection or Compromised Application Server:** Even though the client cannot run arbitrary queries thanks to using an API, we consider the scenario where an attacker somehow injects SQL into a backend query or gets shell access on the server. If the backend is using user-scoped database connections (with RLS active), a successful SQL injection would still be constrained by RLS policies – the attacker’s injected SELECTs or UPDATEs would be subject to the same tenant and user checks. This is a big win: it turns a would-be catastrophic injection (dump all customer data) into a more limited impact (they can only dump the data that the compromised user account itself could access). However, if the server is compromised to the point the attacker can use the service role connection or alter RLS settings, then it’s game over on that node. To mitigate impact: ensure that the database credentials the app uses do **not** have permission to disable RLS or create new roles. In a Supabase setup, the app usually connects with the![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.111.png)  authenticated  or  anon  role via tokens, not a superuser. So even a SQL injection cannot  ALTER TABLE ... DISABLE ROW LEVEL SECURITY because that requires a table owner or superuser (which the app is not using). This principle – *use minimally privileged DB roles for application connections* – contains the blast radius of any injection. Additionally, parameterized queries and ORM query builders should be used extensively to avoid injection in the first place. We also isolate the database – direct connections are not exposed to the internet, only via the application or supabase API. That helps because an attacker can’t just connect to the DB with stolen creds unless they also breach network or environment secrets.
- **Cross-Tenant Data Leakage via Side Channels:** Outside of direct querying, we consider if data could leak through logs, caches, or backups. For example, if we use an in-memory cache (like Redis) keyed only by item ID, one tenant could get a cached value that belongs to another. We prevent that by always including tenant context in cache keys (e.g., use ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.112.png) tenantId:itemId as the key pattern). Another side-channel: logs. We ensure logs don’t inadvertently store sensitive data without tenant tags. For instance, structure logs to include the user/tenant ID for each request, so we can trace and isolate events. And avoid logging entire data objects indiscriminately. In cloud environments, sometimes one service might inadvertently expose data to another (multi-tenancy at infrastructure level) – e.g., an AWS S3 bucket used for multiple tenants. We mitigate that by using per-tenant storage buckets or strict object naming schemes plus access policies. While these aren’t RLS/DB issues per se, they are part of the threat model in multi-tenant systems [48 ](https://www.thecloudplaybook.com/p/api-gateway-custom-authorizers-for-tenant-aware-security#:~:text=Your%20application%20probably%20has%20tenant,in%20places%20you%20haven%27t%20considered). Since our focus is DB, one specific vector: database ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.113.png)*views* or *functions* could bypass RLS if misconfigured. In Postgres, a SECURITY DEFINER function or a view owned by a superuser will execute without RLS by default. A developer might create a convenience view for reporting that accidentally exposes all tenant data if used with a high-privilege role. To avoid that, we ensure any helper views are created with security\_invoker![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.114.png) (so they respect caller’s rights) or materialized in a safe way [49 ](https://supabase.com/docs/guides/auth/architecture#:~:text=Make%20sure%20any%20views%20you,for%20Auth%20data%20are%20protected). We also avoid using superuser for routine queries (as mentioned). All these practices reduce the chance of cross- tenant bleed-through outside of the intended channels.![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.115.png)
- **Misconfiguration of RLS Policies:** A subtle but real risk is human error in writing the policies. If a policy’s logic is wrong, it could allow something it shouldn’t. For example, a developer might write a policy that uses ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.116.png)![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.117.png) OR incorrectly (like  USING (tenant\_id = auth.jwt() ->> 'tenant\_id' OR auth.role() = 'admin') ) which inadvertently allows an admin to see all tenants’ data if not

paired with a tenant check. The fix would be to ensure even admins are scoped to their tenant unless explicitly intended otherwise. Another misconfig is forgetting to enable RLS on a new table. If a table is created and RLS is not enabled, by default it’s wide open to any ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.118.png) authenticated role (since Supabase’s anon key or auth role could select it without restriction). Our process should **mandate RLS on every new table**. Supabase enables it by default in their dashboard UI, but if using raw SQL migrations, you must remember to ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.119.png) ALTER TABLE ... ENABLE ROW LEVEL SECURITY [50](https://supabase.com/docs/guides/database/postgres/row-level-security#:~:text=RLS%20must%20always%20be%20enabled,schema)[ 51 ](https://supabase.com/docs/guides/database/postgres/row-level-security#:~:text=alter%20table%20). We can enforce this by having a checklist in code reviews or even automated tests that scan for tables![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.120.png)  without  RLS.  For  instance,  a  startup  script  could  query  pg\_tables  and  ensure![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.121.png)

rels.rcrelhaspolicy  is true for tables in the public schema. Similarly, adding a new policy might conflict with an existing one. We need to test policies thoroughly: Supabase provides a nice policy tester where you can simulate a JWT and run queries [52](https://supabase.com/docs/guides/auth/oauth-server/token-security#:~:text=1)[ 53 ](https://supabase.com/docs/guides/auth/oauth-server/token-security#:~:text=%22sub%22%3A%20%22test). We would incorporate tests for each policy – e.g., trying to violate the policy and expecting denial. Also, consider ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.122.png)*order of policy evaluation![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.123.png)*  and Postgres versions: earlier versions required careful use of  WITH CHECK  and separate policies per command; Postgres 15 introduced  AS RESTRICTIVE . We should stay updated on best practices (the MakerKit reference suggests patterns and even suggests using![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.124.png)

security definer functions for complex multi-table checks to avoid mistakes [54 55 ](https://makerkit.dev/blog/tutorials/supabase-rls-best-practices#:~:text=Use%20security%20definer%20functions%20when%3A)). There is also a risk of ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.125.png)**performance misconfiguration** being mistaken as a functionality problem: e.g., if a policy is too slow, someone might be tempted to disable RLS “temporarily” for a query. Instead, we address performance by proper indexing and query design (as discussed), so there’s never a need to turn off security for speed. 

- **Privilege Escalation:** This could occur if a user can somehow obtain a higher role or access than intended. One vector for this is if the membership management endpoints or queries are insecure. For example, if there’s an API to add a user to a tenant or change roles, and it’s not properly protected, a malicious user could make themselves an admin. We counter this by heavily guarding those operations: only tenant owners or a central system admin can invite or change roles, and these actions are verified at multiple levels. At the DB level, the ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.126.png) membership table would have RLS policies such as **“only tenant admins can insert or update membership rows for their tenant”**, and perhaps even more granular: *owners* can grant admin, but admins can only invite lower roles, etc., depending on rules. Also, foreign keys from membership to roles can ensure you can’t assign a role that doesn’t exist or that you’re not allowed to assign. Another escalation path could be using one’s privileges in unintended ways – e.g., an admin of Tenant A somehow gaining access to Tenant B by exploiting a lack of check in a cross-tenant feature (like sending a share link of a report and the system failing to validate tenant boundaries). Prevention is again via consistent use of tenant scoping at every layer. Cross-tenant features (if any) should be treated as extremely sensitive – often it’s easier to forbid them entirely. In a strict isolation model, there is no legitimate reason for any user to access another tenant’s data, so we don’t build features that violate that. Multi-tenant integration (like an admin viewing multiple subsidiaries) should still be implemented as that admin having memberships in each tenant, not bypassing the model. Finally, **internal privilege escalation** is considered: e.g., a developer or DBA abusing access. That’s more an organizational control issue – but we can use the database to help by granting developers only the roles they need (maybe they can’t disable RLS or view production data without going through a secure jump host), and using audit logs to detect any out-of-policy access.
- **Migration and Deployment Risks:** When deploying changes to the database, we have to be careful not to inadvertently drop policies or open windows. For example, if we need to backfill data or restructure a table, sometimes RLS might be turned off during migration then back on. During that

window, if the app isn’t in maintenance mode, data leaks could occur. Our deployment process should either apply migrations in a transaction (Postgres allows altering policies transactional) or ensure the application is quiescent if RLS is off. Ideally, we avoid disabling RLS at all – many data migrations can be done tenant-by-tenant or using the service role internally without exposing the data to users. If we *must* disable a policy to copy data, we do it in a controlled script not accessible to end users. Another risk is new tables or columns introduced without policies. We address this by always thinking “authorization” as part of the database schema migration. A migration adding a new table should in the same script add ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.127.png)![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.128.png) ENABLE RLS and at least a basic  BLOCK ALL policy (an empty policy means no access) until we explicitly open it. This way, there’s never a period where a table exists but isn’t protected. Similar care is needed for new columns if they carry sensitive info – though RLS works at row level, we might need column restrictions (Postgres 15+ has ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.129.png) ALTER TABLE ... FORCE ROW LEVEL SECURITY![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.130.png) which ensures even owners obey RLS, and column-level privileges to restrict certain columns). For instance, for PCI data like card numbers, we might store them encrypted and only allow decryption via a secure function that checks user’s role. The threat is if someone accidentally allowed selecting the raw card data – our architecture would avoid that by design (not even give direct select on that column, only via a UDF that maybe masks it unless conditions are met).

In summary, the combination of measures in our design addresses the **OWASP Top 10**  categories for authorization: broken access control, least privilege, etc., and also meets financial industry threats like insider misuse and audit traceability. By analyzing these threats, we’ve chosen design elements to mitigate them: *token-based context to limit identity scope, RLS to enforce that scope in every query, careful handling of powerful credentials, and multi-layer checks to catch errors*. The system is by no means “set and forget” – we will continuously test it. For example, we’ll conduct cross-tenant penetration tests (simulate a user trying to get another tenant’s data) to verify that all such attempts fail (they should be blocked by RLS at minimum)

[39 ](https://mergify.com/blog/application-vs-database-where-should-permissions-live#:~:text=RLS%20should%20not%20be%20treated,leakage%20caused%20by%20missing%20filters). We’ll also monitor for any RLS policy errors in logs (Supabase’s policy tester and our own integration tests where we intentionally try forbidden actions to ensure they’re forbidden). The goal is that any single failure (a bug, a leaked token, etc.) does not result in a full-blown breach because other layers compensate. This layered approach is aligned with security best practices and compliance requirements – e.g., PCI-DSS requires strong access control and separation of duties, which our architecture achieves by not relying on one control alone.![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.131.png)

7. **Final Reference Architecture**

Bringing it all together, we propose a **complete multi-tenant authorization architecture** suitable for an ERP/financial SaaS. It integrates identity management, application logic, and database security in a cohesive way:

**Architecture Components:**

- **Supabase Auth (or equivalent IdP)** – Manages user identities, authentication flows, and issues JWT tokens with the necessary claims (at least a user identifier and an indication of roles/tenant). Acts as the root of trust for who the user is. It supports external OAuth providers and SSO, meaning users can log in via Google, etc., but once authenticated, every user session is represented by our JWT format. For compliance, this layer can enforce password policies, 2FA/MFA, account lockout, and so forth. Supabase Auth uses the Postgres database (auth schema) to store users, which means user records are part of our database – enabling things like foreign key relationships from business data to ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.132.png) auth.users (linking records to their owner user).
- **Backend Server / API Gateway** – This is the application layer that receives client requests. In a pure Supabase setup, this could be the PostgREST auto-generated API or a GraphQL endpoint, plus maybe some cloud function endpoints for custom logic. In a more traditional setup, this is a **Rust backend server** (since the team uses Rust) exposing REST/gRPC endpoints. The important part is that this layer **accepts the JWT from clients and passes it to the database**. If using PostgREST, it automatically does this via the ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.133.png) Authorization: Bearer <JWT> header and mapping to the role. In a custom server, the backend would validate the JWT signature (to ensure it’s valid and not expired) and then use it in one of two ways for DB queries: 
- Use a database driver that supports JWT auth (some clients can exchange the JWT for a session role on Postgres), or 
- Manually execute  SET LOCAL request.jwt.claims = '<jwt-payload>' on the DB connection and ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.134.png) SET LOCAL role = authenticated before running queries [52](https://supabase.com/docs/guides/auth/oauth-server/token-security#:~:text=1)[ 53 ](https://supabase.com/docs/guides/auth/oauth-server/token-security#:~:text=%22sub%22%3A%20%22test). The backend can also implement additional logic on top of DB results. For example, it might combine data from multiple queries, enforce business workflows, or simply forward queries from the frontend to the DB. Crucially, it does *not* implement its own data filtering beyond what’s needed – it relies on the DB for that – but it may implement higher-level checks (like “if user role is not admin, return error 403 before even touching DB” for certain admin-only operations – this check is redundant with DB policy, but provides quicker feedback and clearer error messages).
- **Postgres Database with RLS** – The heart of the system, storing multi-tenant data and enforcing security policies. The schema is designed with tenant isolation in mind (as discussed in Section 4). Row Level Security is enabled on all tables in the **public schema** (and any other schema we expose to the app)![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.135.png) [56](https://makerkit.dev/blog/tutorials/supabase-rls-best-practices#:~:text=Supabase%20exposes%20your%20PostgreSQL%20database,table%20in%20your%20public%20schema)[ 8 ](https://makerkit.dev/blog/tutorials/supabase-rls-best-practices#:~:text=,your%20database%20to%20untrusted%20clients). The database has at least three roles:  anon  (unauthenticated, extremely limited, maybe can only call a signup function), ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.136.png) authenticated (regular user sessions, subject to RLS), and  service\_role (full access, no RLS, used only internally). Additional roles can exist for special cases (like read-only replicas or maintenance). The RLS policies cover every table that contains sensitive data, typically referencing the JWT claims via ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.137.png)![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.138.png) auth.uid() and  auth.jwt() as shown earlier. Some tables (like a ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.139.png) tenant table or a  membership table) may have more complex policies to ensure only tenant admins can see or modify certain information. The **database also contains security extensions![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.140.png)**: for example, we enable  pgcrypto  or use Supabase’s built-in encryption for storing sensitive fields (like hashing PII, encrypting things like SSNs or credit card numbers at rest). We may use ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.141.png) pgAudit to log queries that access particular tables, which is useful for compliance audits (e.g., we could log every SELECT on a payments table, including which user’s token did it). The DB serves as the single source of truth for both data and permission rules.
- **Internal Services** – (Optional) These are trusted backend jobs or microservices that perform tasks like sending emails, generating reports, syncing to external APIs, etc. They need access to data across tenants or at a system level. In our architecture, internal services *never directly use end-user JWTs*. They either:
- Use the service role (bypassing RLS) for tasks that legitimately require cross-tenant access (e.g., compiling a global report). These queries are run in a controlled environment and often read-only. We tightly restrict when and how this is used (maybe only on isolated infrastructure).
- Or use a “system JWT” approach: for example, issue a JWT that has a special claim (like  "sub": "system-user", "role": "worker\_role", "tenant\_id": null![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.142.png)![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.143.png) ). Then define RLS policies to allow  worker\_role to read necessary data. This way, even internal tasks are governed by RLS (just with more permissions). This is more secure but requires writing extra policies. For simplicity, many choose the first approach for internal tasks but keep those tasks out of reach from external entry. Regardless, internal services abide by the principle that they either operate with full clearance but in isolation, or with scoped clearance similar to a privileged user. They also log their actions (so we can trace if an internal job touched tenant data).
- **Client Applications** – This includes the web frontend, mobile apps, etc. Clients obtain JWTs after authentication and attach them to all requests. They should treat JWTs as secrets (never exposing them to third-party scripts). The frontend also usually implements client-side checks for a better UX: e.g., if the user is not in an admin role, it won’t even show admin pages. However, these are purely for UX/convenience – we assume the client can be compromised, so we do not rely on any client enforcement for security. Clients simply pass user input to the backend and display data; they trust the backend/DB to enforce rules.

![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.144.png)

*Schematic of the final reference architecture for multi-tenant SaaS authorization. The User (Client) logs in via Supabase Auth, obtaining a JWT. The client then makes API calls including that JWT. The Backend Server or API Gateway (which could be a custom Rust server or Supabase’s own PostgREST) validates the JWT and uses it when querying the Postgres DB. The database has RLS policies that ensure each query result is scoped to the user’s tenant and permissions. In the dashed red path, an Internal Service  (running with elevated privileges) communicates with the database using a special service role or token, bypassing RLS or using special policies – this is used for admin tasks or cross-tenant operations that are not exposed to end-users. All data flows are thus either constrained by RLS (normal user paths) or occur in a controlled, internal context (service paths). This dual* *approach  supports  both  multi-tenant  isolation  and  the  needs  of  background  processes,  without  ever compromising the security of tenant data.*

**Policy Examples in Action:**  To illustrate, consider how a request flows through this architecture for a concrete action – say, “User from Tenant X attempts to view an invoice”: 

1. The user’s frontend calls  GET /invoices/123 with their JWT. The JWT includes  sub = user123![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.145.png)![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.146.png)![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.147.png)

and  tenant\_id = X .

2. The backend receives this. Suppose this endpoint is open to all authenticated users, so the backend simply forwards the query to the DB (e.g., ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.148.png) SELECT \* FROM invoices WHERE id=123 ). It might attach the JWT in a header or set the session context accordingly.![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.149.png)
2. In Postgres, because the role is  authenticated and RLS is enabled on  invoices , the RLS policy we created earlier ( USING (tenant\_id = auth.jwt()->>'tenant\_id') ) kicks in[ 14 ](https://supabase.com/docs/guides/auth/oauth-server/token-security#:~:text=,is%20in%20the%20token). Postgres transparently adds ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.150.png) AND invoices.tenant\_id = 'X' to the query. If invoice 123 belongs to

tenant X, the row is returned; if it belongs to tenant Y, Postgres returns 0 rows (or an error if using SELECT ... FOR UPDATE, etc.).

4. The backend gets the result. If 0 rows, it can return 404 Not Found to the user – without revealing that the ID was valid but in another tenant. (This avoids an Oracle that could be abused to guess IDs.) ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.151.png)![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.152.png)
4. If the row was returned and, say, contained sensitive fields like  amount and  status , the backend might still apply additional formatting or checks. Maybe only users with “finance” role should see amount![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.153.png)![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.154.png) . If the JWT had a claim for role, the backend could strip out  amount field for

unauthorized roles (defense in depth, again the DB could also have used column-level SELECT privileges).

6. The user receives the invoice data permitted for them.

Now consider an **admin service** generating a company-wide financial report: it might connect with the service role and do a ![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.155.png) SELECT SUM(amount) FROM invoices across tenants. That query bypasses RLS, so it will sum everything. We ensure this service and query are running in a secure environment where only

authorized staff or processes can invoke it (and audit the usage). If we wanted, we could instead write a special function in the DB for reports that uses definer rights to gather data but only exposes aggregated results – however, if it’s truly cross-tenant aggregate, it might not violate individual tenant privacy (since it’s aggregated). Still, being mindful, we would probably not expose cross-tenant aggregates to any user accounts unless it’s a global admin.

**ERP-Grade Considerations:** Financial systems demand not just security but accuracy and traceability. Our architecture supports that by: -![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.156.png) **Ensuring no unauthorized modifications**  – RLS with  WITH CHECK prevents, for example, a user from inserting a transaction under another user’s account or altering fields they shouldn’t. We could complement this with database triggers for extra validation (e.g., preventing negative invoice amounts, etc., though that’s business logic). - **Audit trails** – Because every query carries the  user’s  ID  via  JWT,  we  can  log  who  did  what.  For  instance,  an  update  trigger  can  record![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.157.png)

current\_setting('request.jwt.claims') ->> 'sub'  into an audit table, giving us a per-row change history with the responsible user. This is crucial for compliance (PCI and others often require tracking data access and changes). - **Compliance isolation** – In some cases, certain data may need to be isolated even among roles (for example, encryption keys or personal data might be accessible only to a subset of roles). We can use a combination of RLS and Postgres row/column permissions to achieve **need- to-know access**. For example, perhaps only a “ComplianceOfficer” role can see full credit card numbers –

RLS can ensure  (auth.jwt() ->> 'role' = 'ComplianceOfficer') on the card info table, so even if a normal admin tries, they get nothing. This can satisfy PCI’s requirement that only users with specific job function can view cardholder data. Those special roles would not be assignable via the normal tenant admin; they’d be global roles assigned by system administrators to compliance team members.![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.158.png)

- **Encryption** – While not directly an auth matter, we note that using column encryption or encryption at rest for sensitive fields can add another layer. Supabase has an extension for encryption![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.159.png)

( pgsodium , though as of writing might be in preview). Alternatively, store encrypted blobs and only decrypt in app when needed. This way, even if someone bypassed RLS (say a misconfigured policy), the data they see might still be gibberish without the decryption key. This is an extra safeguard for things like SSNs or PANs in databases.

**Final Thoughts:** This reference architecture leverages **standard open-source technologies** (Postgres, JWT, RLS) as provided by Supabase, which avoids reinventing wheels. The use of JWT means the system is stateless at the app layer (the token carries the session info), making it scalable. Postgres itself can handle a large number of tenants in one database (with proper partitioning or indexing strategies if needed, though those are transparent to the auth logic). By isolating tenants in the core data access layer, we greatly reduce the risk of the most damaging breach: one customer seeing another’s confidential data. Each layer – auth, app, DB – is configured to perform the security checks it’s best at, and together they form a robust mesh of defenses. 

This approach is already being successfully used in real multi-tenant products (as evidenced by community examples like the LockIn case [57](https://dev.to/blackie360/-enforcing-row-level-security-in-supabase-a-deep-dive-into-lockins-multi-tenant-architecture-4hd2#:~:text=Introduction)[ 58 ](https://dev.to/blackie360/-enforcing-row-level-security-in-supabase-a-deep-dive-into-lockins-multi-tenant-architecture-4hd2#:~:text=Critical%20Requirement%3A%20Complete%20data%20isolation,level%20security)and Mergify’s experience [59](https://mergify.com/blog/application-vs-database-where-should-permissions-live#:~:text=Obviously%2C%20the%20experience%20raised%20an,database%20with%20more%20optimized%20queries)[ 60 ](https://mergify.com/blog/application-vs-database-where-should-permissions-live#:~:text=database%20schema%20was%20already%20solid,and%20we%20made%20the%20switch)). It is well-suited for ERP/financial systems where correctness and security trump raw convenience. Yes, it requires careful implementation![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.160.png)![](92132fdc-85a1-4bfb-af83-3c2cdb4d7290_00000.161.png)

and vigilant maintenance (policies need updates as features grow, and developers must be trained on RLS and tenant-aware coding). But the payoff is significant: a system that can confidently segregate and protect each tenant’s data, meeting high standards like PCI-DSS compliance. Even under sophisticated attacks or inadvertent mistakes, the layered controls ensure there’s no single point of failure in authorization. Each request either gets precisely the data it’s allowed, or nothing at all – and that is the hallmark of a secure multi-tenant SaaS architecture.
