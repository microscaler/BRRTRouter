**High-Security Multi-Tenant Auth & AuthZ Architecture**

1. **OAuth 2.1 / OIDC Flow Design**

**Authorization Code (PKCE) for Users:** All human users authenticate via the OAuth 2.1 Authorization Code flow with PKCE, ensuring no tokens are exposed to the browser address bar. OAuth 2.1 formalizes security best practices – PKCE is mandatory for all clients (including web apps) to prevent interception of auth codes

[1](https://www.descope.com/blog/post/oauth-2-0-vs-oauth-2-1#:~:text=,like%20implicit%20and%20password%20flows)[ 2 ](https://www.descope.com/blog/post/oauth-2-0-vs-oauth-2-1#:~:text=PKCE%20is%20now%20required), and legacy implicit or password grants are removed[ 3 ](https://www.descope.com/blog/post/oauth-2-0-vs-oauth-2-1#:~:text=Implicit%20grant%20removed). In practice, this means a user logging into![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.003.png)![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.004.png)

the SaaS (or an on-prem instance) is redirected to the central Identity Provider (IDAM) for login, MFA, and SSO. The IDAM could be a dedicated service (e.g. Keycloak in self-hosted mode) or a cloud provider (Auth0, Okta, etc.) configured for the SaaS tenant. The IDAM authenticates the user (including MFA challenges if enabled) and obtains consent if required (for example, consenting to allow a third-party integration). It then issues an **Authorization Code** which the client app exchanges for tokens (ID token for user identity, Access token for API calls). We enforce **exact redirect URI matching and refresh token rotation** as per OAuth 2.1 recommendations[ 4 ](https://www.descope.com/blog/post/oauth-2-0-vs-oauth-2-1#:~:text=,attacks%20across%20all%20app%20types)– this limits reuse of stolen refresh tokens and ensures long-lived sessions can be revoked by the IDAM.![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.005.png)

**Sequence – User Login Flow:** The following sequence diagram shows the end-to-end Authorization Code flow with PKCE and token exchange to the app. This includes SSO and MFA handled by the IDAM (e.g. Okta/ Keycloak), though those sub-steps are abstracted for brevity:

sequenceDiagram![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.006.png)

`    `participant User

`    `participant Browser

`    `participant AuthServer as IDAM (Auth Service)

`    `participant App as SaaS Backend

`    `User ->> Browser: Navigate to SaaS application

`    `Browser ->> AuthServer: \*\*OAuth2 Authorization Request\*\* (Code + PKCE challenge)

`    `AuthServer -->> Browser: Prompt user for login (password/MFA/SSO)

`    `AuthServer -->> Browser: \*\*Redirect back\*\* with Authorization Code

`    `Browser ->> AuthServer: \*\*Token Request\*\* (Code + PKCE verifier)

`    `AuthServer -->> Browser: \*\*ID Token + Access Token\*\* (JWTs)

`    `Browser -->> App: Invoke SaaS API with Access Token (in Authorization header)

`    `App ->> AuthServer: \*\*Verify JWT\*\* (signature & claims via JWKS or introspection)

`    `App -->> User: Process request (user is authenticated)

In this flow, the IDAM issues JWTs that the application will validate on each request (by verifying the signature and expiration, and optionally checking with IDAM if token is active). We prefer **short-lived access tokens (5–15 min)** with one-time use refresh tokens to reduce the window for abuse. Any refresh token misuse triggers revocation of that session. The IDAM should support **OIDC** so that an ID token (JWT) provides user profile claims (name, email, etc.) and the access token carries authorization info (scopes or permissions). 

**Client Credentials for Services:**  For machine-to-machine interactions (internal microservice calls or external integrations), we use the OAuth 2.1 **Client Credentials** grant. Each non-human client (service or third-party app) is provisioned with its own credentials (client ID and secret, or certificate/JWT for auth) in the central IDAM. By default, these service credentials are scoped to only the minimal APIs needed. For example, an “Invoice Processor” service might get a token with scope ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.007.png) invoice.write to call the Billing API, but it would not have user-specific scopes. The IDAM issues **JWT access tokens** for services as well, which include a service principal identity and permitted scope. Every token is tagged with an **audience** (the intended API or resource) to prevent reuse against a different service. This means Service A must request a token specifically for Service B’s audience; Service B will reject tokens not intended for it. By scoping service tokens per audience, we **prevent “god-mode” tokens**  from roaming the ecosystem – if Service A is compromised, its token cannot directly call Service C unless explicitly allowed and obtained. Real-world analogy: Auth0 and Okta let you define multiple APIs with distinct audiences and issue client tokens per API. Keycloak’s implementation similarly allows exchanging a token for one client to a token for another within the realm[ 5](https://www.keycloak.org/securing-apps/token-exchange#:~:text=The%20capabilities%20of%20Keycloak%20for,token%20exchange%20are%20as%20follows)[ 6 ](https://www.keycloak.org/securing-apps/token-exchange#:~:text=1,client%20in%20the%20same%20realm). ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.008.png)

**OAuth2 Token Exchange for Delegation:** To support microservice-to-microservice calls on behalf of a user, we employ the OAuth 2.0 Token Exchange (RFC 8693) protocol. This allows a service to exchange an incoming user token for a new token that is *down-scoped* and targeted to the next service in the call chain![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.009.png)

[7](https://docs.pingidentity.com/pingam//7.2/oauth2-guide/oauth2-token-exchange.html#:~:text=Exchanged%20tokens%20are%20restricted%20when,many%20restricted%20tokens%20as%20required)[ 8 ](https://docs.pingidentity.com/pingam//7.2/oauth2-guide/oauth2-token-exchange.html#:~:text=Clients%20may%20want%20to%20exchange,for%20impersonation%20and%20delegation%20purposes). Crucially, the exchanged token can carry an **“actor” claim** ( act ) identifying the original calling![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.010.png)

service for transparency[ 9](https://docs.pingidentity.com/pingam//7.2/oauth2-guide/oauth2-token-exchange.html#:~:text=About%20the%20act%20claim)[ 10 ](https://docs.pingidentity.com/pingam//7.2/oauth2-guide/oauth2-token-exchange.html#:~:text=The%20,actor%20token%20used%20during%20delegation). For example, if a user calls **Service A**, which then needs to call **Service B**, Service A will present the user’s access token to the IDAM’s Token Exchange endpoint. It authenticates with its own credentials (proving Service A is allowed to act) and requests a new token for ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.011.png)**Service B** with only the subset of permissions needed. The IDAM validates that Service A is permitted to delegate for this user (potentially using a policy like a ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.012.png) may\_act constraint [11 ](https://docs.pingidentity.com/pingam//7.2/oauth2-guide/oauth2-token-exchange.html#:~:text=the%20party%20or%20parties%20allowed,see%20About%20the%20may_act%20claim)) and then returns a token for Service B. This new JWT might include an ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.013.png)![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.014.png) act claim indicating it was issued to Service A acting on behalf of user X. **Service B** will validate two things: (1) the JWT is intended for **Service B** (audience check), and (2) the call itself is coming from an authenticated Service A (more on internal auth in section 3). This pattern ensures *least privilege* propagation of user identity – Service A doesn’t forward the user’s original broad token (which might have had access to more than Service B should get) [12](https://docs.pingidentity.com/pingam//7.2/oauth2-guide/oauth2-token-exchange.html#:~:text=To%20impersonate%20means%20to%20pretend,the%20user%20and%20the%20client)[ 13 ](https://docs.pingidentity.com/pingam//7.2/oauth2-guide/oauth2-token-exchange.html#:~:text=If%20the%20user%20only%20wants,scope). Instead, it gives Service B a narrowed token (e.g. only ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.015.png)![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.016.png) transfer scope in the Ping Identity example of a banking app [14 ](https://docs.pingidentity.com/pingam//7.2/oauth2-guide/oauth2-token-exchange.html#:~:text=When%20the%20user%20logs%20in%2C,can%20use%20in%20the%20app)). Real implementations of token exchange are seen in systems like ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.017.png)**Keycloak** (which supports a token-exchange grant for cross- client tokens) and **Zitadel** or **Curity** STS. Even Auth0 can be extended to do token exchange via rules or the Auth0 Pipeline, though not out-of-the-box. The important point is that RFC 8693 provides a standard, secure way to handle user delegation in microservices [15 ](https://www.scottbrady.io/oauth/delegation-patterns-for-oauth-20#:~:text=OAuth%202)rather than ad-hoc methods.![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.018.png)

**Sequence – Service-to-Service Delegation:**  The diagram below shows **Service A**  calling **Service B**  on behalf of a user, using mTLS and token exchange to enforce trust and least privilege:

sequenceDiagram![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.019.png)

`    `participant A as Service A (API)

`    `participant STS as IDAM Security Token Service

`    `participant B as Service B (API)

`    `Note over A,STS: \*Service A exchanges user’s token for a B-specific token\*     A ->> STS: \*\*Token Exchange\*\* (grant\_type=token-exchange, aud=ServiceB, subj\_token=userJWT)

`    `STS -->> A: \*\*Down-scoped JWT\*\* for Service B (incl. user claims, limited scope)

`    `Note over A,B: \*A now calls B with a token that B trusts and has minimal scope\*

`    `A ->> B: \*\*mTLS request\*\* + JWT (aud=ServiceB)

`    `B ->> B: Verify mTLS client identity (SPIFFE ID for Service A)

`    `B ->> STS: Validate JWT (signature, exp, aud=ServiceB, scope)

`    `B -->> A: Return response (authorized as user X via A)

In this flow, Service A never passes along the original user token – it trades it for a new one that *cannot be used* except for calling Service B. If Service A were compromised, the attacker would only obtain tokens valid for specific downstream services (and often short-lived). As a rule, any service exchanging a token must authenticate itself (here via its own credentials or mTLS certificate), so the STS (Security Token Service) knows the actor. The returned token includes the ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.020.png)**acting party claim** ( act.sub ) to record the delegation![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.021.png)

[9](https://docs.pingidentity.com/pingam//7.2/oauth2-guide/oauth2-token-exchange.html#:~:text=About%20the%20act%20claim)[ 10 ](https://docs.pingidentity.com/pingam//7.2/oauth2-guide/oauth2-token-exchange.html#:~:text=The%20,actor%20token%20used%20during%20delegation). Service B, upon seeing an  act claim, can log that “Service A acted on behalf of User X,” providing![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.010.png)

an audit trail. (If we had used a pure impersonation flow, the token would look like User X directly and B wouldn’t know a delegation happened![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.022.png) [16 ](https://docs.pingidentity.com/pingam//7.2/oauth2-guide/oauth2-token-exchange.html#:~:text=Impersonated%20tokens%20cannot%20be%20told,is%20different%20for%20delegated%20tokens), so we generally favor delegation with  act for internal calls.)![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.023.png)

**Down-Scoping & Audience Restriction:**  The token exchange approach inherently produces a **down- scoped access token with an audience restriction**. “Down-scoping” means the new token has equal or fewer privileges than the original [17](https://docs.pingidentity.com/pingam//7.2/oauth2-guide/oauth2-token-exchange.html#:~:text=As%20a%20high,have%20been%20expanded%20or%20restricted)[ 13 ](https://docs.pingidentity.com/pingam//7.2/oauth2-guide/oauth2-token-exchange.html#:~:text=If%20the%20user%20only%20wants,scope). For example, our IDAM might allow Service A to request a token for Service B that has only a single permission (say, ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.024.png)![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.025.png) orders:read ) even if the user’s original token to A

had broad permissions ( orders:read ,  orders:write ,  inventory:all , etc.). The audience claim![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.026.png)

( aud ) in the new JWT will be set to Service B, so even if Service A mistakenly tried to use that token on Service C, Service C would reject it. This design confines tokens to their intended scope and audience [18 ](https://www.keycloak.org/securing-apps/token-exchange#:~:text=,permissions%20or%20scopes). It is directly inspired by real-world practices: for instance, ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.027.png)**Google Cloud** allows down-scoping OAuth tokens

for specific GCS buckets; and AWS’s Cognito can map an access token to limited IAM credentials for a specific resource [19](https://aws.amazon.com/blogs/apn/tenant-switching-and-custom-permissions-in-a-multi-tenant-serverless-application/#:~:text=The%20IAM%20policy%20contains%20a,identity%20pool%20is%20configured%20with)[ 20 ](https://aws.amazon.com/blogs/apn/tenant-switching-and-custom-permissions-in-a-multi-tenant-serverless-application/#:~:text=,ID%20of%20user%E2%80%99s%20active%20tenant). ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.028.png)

In summary, our OAuth 2.1/OIDC design uses **battle-tested flows**: Auth Code + PKCE for user login (with optional SSO/MFA), Client Credentials for headless integrations, and Token Exchange for any multi-hop delegation. All tokens are JWTs (details in next section) issued by a central authority (the IDAM/STS), and every request is authenticated either by a user token or a service token. Implementations like **Keycloak** can cover all these capabilities in one (with support for OIDC, SAML SSO, user federation, MFA, and even token exchange with proper client permissions). Auth0/Okta can be used in cloud deployments – they similarly support Auth Code and Client Credentials, and Auth0 has an RBAC system where you can map roles to API scopes for tokens. **Google Identity** (Firebase Auth or GCP IAM) showcases the principles too: e.g. Firebase issues short JWTs with limited roles that integrate with its services’ rules, and GCP’s service accounts issue tokens with specific scopes to access APIs. We favor solutions that are **standards-compliant**  (OIDC, OAuth2.1) so that switching out the backing IDAM (e.g. from Auth0 to an on-prem Keycloak for a customer installation) does not require code changes – only configuration.

2. **JWT Design for SaaS**

JSON Web Tokens (JWTs) are the core **authn credential** in this architecture – they carry user or service identity and entitlements between components. We design our JWT payload and usage with multi-tenant security and scalability in mind.

**Key Claims and Tenant Scoping:** Each JWT issued by the IDAM includes standard claims and custom claims to ensure it’s *tenant-aware* and least-privilege. Important claims include: 

- **iss (Issuer):** URI of the IDAM (e.g.  https://auth.saas.com/ ). This lets services verify the token’s origin and fetch the signing keys (from the JWKS endpoint at ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.031.png)![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.032.png)![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.033.png) .../.well-known/ jwks.json )[ 21 ](https://supabase.com/docs/guides/auth/jwts#:~:text=Claim%20Description%20,access%20to%20profile%20information%20without).![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.034.png)
- **sub (Subject):** The principal identifier. For a user JWT,  sub might be a UUID for the user *within a tenant*. To guarantee uniqueness across tenants, one strategy is to embed the tenant ID in the sub (e.g. ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.035.png) user123@tenantA or a composite GUID). Alternatively, use a separate claim for tenant and keep ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.036.png) sub globally unique (like  auth0|abc12345 style) – but **either way the tenant context must be derivable**. For service account tokens,  sub could be a service principal ID.
- **tenant (Tenant ID):** An explicit tenant identifier claim. This is critical for multi-tenancy: every token says “this token is for tenant ABC Corp”. All resource servers will enforce that this claim matches the tenant context of the request (e.g. the tenant in the URL or database query). In AWS Cognito, one might use a custom attribute or group to hold tenant ID [20 ](https://aws.amazon.com/blogs/apn/tenant-switching-and-custom-permissions-in-a-multi-tenant-serverless-application/#:~:text=,ID%20of%20user%E2%80%99s%20active%20tenant); in our system, we treat it as first-class. For instance, if token ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.037.png)![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.038.png) sub = 56e7890 (user id) and  tenant = acme-co , Service B will only return data from  acme-co and reject any attempt to use that token for another tenant.![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.039.png)
- **aud (Audience):** The intended service or resource for the token. We issue **service-specific audiences** to prevent tokens from being accepted where not intended. For example, a JWT obtained to call the “Billing API” will have ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.040.png) aud: billing-api – other services should reject it. (In OIDC

terms, some tokens may use an  azp or authorized party, but we’ll use  aud for simplicity).![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.041.png)

- **iat and  exp (Issued At, Expiration):** We set relatively short expiration times (e.g. 5-15 minutes for access tokens). This ensures that if a user’s roles change or if a token is stolen, it won’t live long. Clients are expected to use refresh tokens to get new JWTs after expiration.![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.042.png)![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.043.png)![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.044.png)![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.045.png)
- **jti (Token ID):** Each JWT can carry a unique identifier. We use  jti for revocation when needed – e.g. if a token must be explicitly revoked prior to expiry (perhaps user left the company, or a refresh token misuse detected), the ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.046.png) jti goes on a revocation list that resource servers check (more on revocation later).![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.047.png)![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.048.png)
- **scope or  permissions :** Rather than embedding high-level roles, JWTs carry fine-grained scopes or permission claims. For instance, a token might have ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.049.png)

"scope": "invoice.read invoice.pay" . Each microservice will interpret scopes relevant to its domain. This aligns with OAuth 2.0 practice and avoids overloading the token with entire role definitions. If needed, we can include a list of explicit permissions in a custom claim, say ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.050.png) "perm": ["invoice:create", "invoice:pay"]![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.051.png) , especially for on-prem where offline introspection is not feasible. A *minimal* set of permissions is included – **only those the target service needs to know about**.

**Multi-Tenant-Safe JWT Payload Example:** An access token for user Jane of tenant Acme Corp, calling the Pricing service, might look like:

"sub": "a1b2c3d4-e5f6-... (Jane's user UUID)", "tenant": "acme-corp",

"aud": "pricing-api",

"iat": 1700000000,

"exp": 1700000300,

"scope": "pricing:read pricing:quote",

"act": {

"sub": "service-123",

"iss": "https://idam.saas.com/"

}

}

Here, Jane’s token has two scopes related to the Pricing API. It also shows an  act claim: this would appear if, for example, this token was minted via a delegation (perhaps a background service acting for Jane) – the![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.053.png)![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.054.png)![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.055.png)

act.sub identifies the service principal that is the acting party[ 9 ](https://docs.pingidentity.com/pingam//7.2/oauth2-guide/oauth2-token-exchange.html#:~:text=About%20the%20act%20claim). In normal user tokens,  act would be absent.![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.056.png)

**Why Not Roles in JWTs:** We deliberately avoid putting full role names or large group lists in the JWT. Instead of ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.057.png) "roles": ["Admin", "Auditor", ...] , we favor explicit permissions or scopes. There are several reasons: 1. **Roles can be numerous and dynamic:** In a complex app, a user might accumulate dozens of roles – including cross-tenant roles – which would bloat the token size and potentially exceed header size limits. (Azure AD, for instance, caps tokens to 200 group claims; beyond that it omits them and provides a Graph API link instead [22 ](https://securecloud.blog/2020/05/14/deep-diver-azure-ad-groups-roles-claims-for-developers-and-it-pros-with-code-examples/#:~:text=,group%20names%20besides%20group%20id%E2%80%99s). This shows how too many roles don’t fit in a JWT.) Large tokens hurt performance and may be rejected by proxies. 2.![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.058.png) **Stale Authorization:**  Roles are an indirection to permissions and can change at any time (e.g. an admin revokes the “Admin” role from a user). If the JWT

holds the old roles, it remains valid until expiry – meaning the user could still access admin endpoints for a few minutes *or hours* unless we actively revoke the token. Permissions in tokens have the same issue, but since we issue very short-lived tokens, the window of staleness is small. Moreover, if a user’s permissions change drastically, we can include a ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.059.png)![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.060.png) ver (version) claim or use  iat to invalidate older tokens when the role assignment changes (by tracking the last role-change timestamp). 3. **Least Privilege & Simplicity:** Embedding roles would require every microservice to know the semantics of each role to enforce authZ, which couples services to the IDAM’s role definitions. Instead, services can simply check for the specific permission scopes they care about (e.g. “does this token have ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.061.png) invoice.pay ?”). This is both simpler and aligns with microservice autonomy. An example from ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.062.png)**Supabase**: Supabase’s JWTs include a  role claim that maps to a Postgres role for RLS [23 ](https://supabase.com/docs/guides/auth/jwts#:~:text=,request%20to%20the%20Auth%20server). But in a more granular permission system, one “role” isn’t enough![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.063.png)

- you often need per-action flags. Our approach would put those flags in the token or rely on an external check.

Therefore, **we do not put a full ACL or role list in the token**. If a user has many roles, the IDAM can compute the union of permissions and include those, or simply issue a reference token for introspection. A hybrid approach could also be used: e.g. include an opaque ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.064.png) perm\_id in JWT and the service does a cache lookup to a permission set.

**Token  Size  vs  Introspection  Trade-offs:**  Using  JWTs  (self-contained  tokens)  means  services  can authenticate without contacting IDAM on each request – which is great for performance and decoupling. However, as mentioned, self-contained tokens carry potentially sensitive or large auth data that can become stale. We mitigate this by: -![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.065.png) **Short TTLs:**  By limiting  exp  to e.g. 5-15 minutes, we accept that role/ permission changes can take that long to fully propagate (which is usually acceptable under PCI/finance guidelines, especially if we log changes). - **Refresh on critical changes:**  For certain high-risk actions, services may demand a fresh token or re-auth (MFA) if the token is older than a few minutes, limiting the impact of staleness. - **Introspection for long-lived contexts:** For on-prem deployments or certain flows (like validating a long-lived API key), we provide an introspection endpoint at the IDAM. A service can POST a token and get back a yes/no or the token’s claims if needed. This is useful for backend services that want to always check if a token was revoked. Because introspection adds latency and a dependency, we use it sparingly – primarily for **opaque tokens**  (like reference tokens or refresh tokens) or for admin-level verifications. - **Distributed cache of revocations:** The IDAM can push a cache of revoked JTIs or invalidated user IDs to the services (or a sidecar). This way, each service, when validating a JWT, can do a quick check (in memory or Redis) to see “Is jti X globally revoked or is user Y’s token version outdated?”. This avoids a direct introspection call on every request but still allows near-real-time revocation.

In summary, we try to keep JWTs **lean and focused** – user ID, tenant ID, necessary scopes, and metadata for validation. This yields tokens on the order of a few hundred bytes, which is easily handled in HTTP headers and fast to parse. By not including long lists of roles or groups, we avoid the scenario where Azure AD had to truncate group claims due to size [22 ](https://securecloud.blog/2020/05/14/deep-diver-azure-ad-groups-roles-claims-for-developers-and-it-pros-with-code-examples/#:~:text=,group%20names%20besides%20group%20id%E2%80%99s). If a user has an extreme number of permissions, we’d likely need to use an opaque token and rely on on-demand lookup, but that’s rare in a well-scoped SaaS (permissions are typically bounded by the number of features).![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.066.png)

**Impersonation & Actor Claims:**  In scenarios where an admin user or support engineer needs to impersonate another user, or a service account impersonates a user, we issue a special JWT with an ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.067.png) act claim (and possibly an ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.068.png)![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.069.png) impersonated flag). The  act claim, as defined in RFC 8693, is a JSON object indicating the acting party[ 9 ](https://docs.pingidentity.com/pingam//7.2/oauth2-guide/oauth2-token-exchange.html#:~:text=About%20the%20act%20claim). For example, if a support service exchanges its admin token for a token as

Alice (to view what Alice sees), the token might include  "act": {"sub": "support-agent-123", 

"role": "SupportEngineer"} . The resource service can decide to allow or restrict certain actions when an![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.070.png)  act  is present (for instance, maybe you can read as the user but not perform state-changing operations). This provides a **transparent chain of custody** in the token itself. We prefer this over pure impersonation (where the token’s subject is just Alice and nothing in the token says it’s an impersonation) because pure impersonation has risk – the receiving service cannot distinguish it from the real Alice [17 ](https://docs.pingidentity.com/pingam//7.2/oauth2-guide/oauth2-token-exchange.html#:~:text=As%20a%20high,have%20been%20expanded%20or%20restricted). However, for internal trusted components, we may allow “naked impersonation” tokens in strictly controlled cases (Keycloak supports an admin impersonation feature, for example). Any impersonation is tightly audited via logs.![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.071.png)

**Revocation Strategies:** Robust revocation is crucial in a high-security environment (consider a stolen token or a user offboarding). We employ multiple strategies in tandem: - **Short Lifetimes:** As emphasized, the primary defense is that tokens expire quickly on their own. This limits the window of misuse. It also means that if we need to **mass-revoke** (say, all tokens issued before a certain time due to a security incident), we can invalidate the signing key and all tokens will fail verification. Our IDAM will regularly rotate signing keys (with overlap) and retire old ones, which also forces periodic re-auth of clients. - **JTI Blacklisting:**  For targeted revocation (e.g. user manually logs out, or an API key is withdrawn), the IDAM will put that token’s![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.072.png)

jti (or a hash of it) in a blacklist store. Each service has a lightweight mechanism (in-memory or cache) to check incoming JWT ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.073.png) jti against recent revokes. Because access tokens are short-lived, this list remains small (and can evict entries after token expiry time). For refresh tokens, which are longer lived, the IDAM keeps a revoke list and won’t issue new access tokens if the refresh is revoked. - **Token Versioning (Soft Revocation):** We maintain a **token version number** or a last invalidation timestamp per user (and per client). This could be implemented by a claim like ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.074.png) "v": 5 in the JWT and a field in the user’s record “currentTokenVersion = 6” if an admin invalidated their sessions. Resource services can either query IDAM for current version if suspicious, or the IDAM can embed a timestamp like ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.075.png) "last\_password\_change": 1700000000![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.076.png)  in the token. For example, Firebase Auth includes  auth\_time  and uses the user’s![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.077.png)

lastLogout to invalidate older tokens – if a token’s  iat is before a stored cutoff, it’s rejected. We can do similarly: e.g. if a user resets their password or an admin hits “log out all sessions”, we update a![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.078.png)![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.079.png)![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.080.png)

revokedBefore timestamp in IDAM. Services then compare token’s  iat to that for that user – if  iat is older, reject it. This approach avoids per-token tracking and is good for sweeping invalidation (at the cost of a quick lookup). - **Refresh Token Rotation & Reuse Detection:** Any refresh token that a client uses to get a new access token is one-time use (rotated). If an attacker somehow steals a refresh token and tries to use it after the legitimate client already did, the IDAM will detect the second use and revoke the session immediately [24 ](https://www.descope.com/blog/post/oauth-2-0-vs-oauth-2-1#:~:text=open%20redirect%20vulnerabilities). This is essential for long-lived sessions (even though refresh tokens might not be heavily used in on-prem scenarios, it’s a must for the hosted SaaS for continuous login). - ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.081.png)**Opaque Tokens for External Parties:** In cases where an external integration needs a token that doesn’t expire quickly (say a server-to-server token for a nightly job), we might issue an opaque token (e.g. a UUID or structured reference). That token must be introspected by our IDAM on each use – this way we have absolute control: we can revoke it server-side instantly. This would be done only for specific use cases (like legacy systems or non-JWT-friendly clients). By default, we stick to JWTs for consistency and performance.

**Real-World Examples & Inspiration: ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.082.png)**

- *AWS Cognito:* Cognito User Pool tokens are JWTs that include a  cognito:groups claim if the user is in any groups (roles) and can include custom attributes like  custom:tenant\_id . One AWS SaaS design uses a **group per tenant**  model 25[ 20](https://aws.amazon.com/blogs/apn/tenant-switching-and-custom-permissions-in-a-multi-tenant-serverless-application/#:~:text=At%20this%20stage%2C%20we%20have,appear%20in%20the%20ID%20token) [ – ](https://aws.amazon.com/blogs/apn/tenant-switching-and-custom-permissions-in-a-multi-tenant-serverless-application/#:~:text=,ID%20of%20user%E2%80%99s%20active%20tenant)the token lists all tenant groups the user belongs to, and a custom:tenant\_id![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.083.png)![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.084.png)  for the current tenant context. This is similar to our approach of having both a membership list and an active tenant indicator. However, as noted, if a user has too many groups, Cognito JWTs can get large and AWS will not include them all, requiring an extra lookup [22 ](https://securecloud.blog/2020/05/14/deep-diver-azure-ad-groups-roles-claims-for-developers-and-it-pros-with-code-examples/#:~:text=,group%20names%20besides%20group%20id%E2%80%99s). by keeping the tenant list small (most users only in their one tenant) and focusing on the active tenant. AWS Cognito also shows how JWTs tie into resource access: it can map token attributes to IAM conditions for DynamoDB, scoping access to rows with that tenant tag [19 ](https://aws.amazon.com/blogs/apn/tenant-switching-and-custom-permissions-in-a-multi-tenant-serverless-application/#:~:text=The%20IAM%20policy%20contains%20a,identity%20pool%20is%20configured%20with). We aim for a similar *defense in depth* – the JWT carries tenant info which is enforced at the application layer (and even database layer via row-level security, if applicable).![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.086.png) ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.087.png)

- *Supabase:* Supabase issues JWTs with a  role claim which the Postgres row-level security uses to grant read/write rights![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.088.png) [23 ](https://supabase.com/docs/guides/auth/jwts#:~:text=,request%20to%20the%20Auth%20server). Our JWTs give services the info to enforce policies internally – e.g. the Pricing service might trust ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.093.png)![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.094.png) tenant and  scope claims in the JWT to decide which records to return.
- *OIDC Providers:* Okta and Auth0 both issue JWT access tokens that can include roles or permissions. This is essentially what we’re doing: each microservice’s API scopes are defined, and the token gets those the user (or service account) is allowed. Our system’s central IDAM will manage that (see section 5). ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.095.png)

In closing, JWTs in our system are *tenant-scoped, least-privilege by design, and ephemeral*. They enable stateless verification, which is crucial for performance in a distributed system, but we layer complementary measures (short life, verifiable signatures, actor tracking, revocation checks) to meet high security standards (PCI compliance, auditability). Every JWT can be traced to who issued it and when, and carries just enough info to authorize the request and no more.

3. **SPIFFE / SPIRE for Internal Identity**

While OAuth2/JWTs handle *who* is calling (user or service account) and *what* they can do, we also need to ensure that *the caller is a legitimate service in our architecture* – not an imposter. This is where **SPIFFE/SPIRE** comes in as our solution for internal workload identity and authentication at the transport level.

**What SPIFFE Solves (Workload Identity & Zero Trust):** SPIFFE (Secure Production Identity Framework For Everyone)  provides  a  standard  way  to  assign  identities  to  workloads  (services,  processes)  and  to authenticate those identities using X.509 certificates or JWT-SVIDs. In essence, SPIFFE + SPIRE (the implementation) answers the question: *“Is the service that’s connecting to me really the genuine Service A from my cluster, or is it someone spoofing or a compromised container?”* OAuth tokens alone can’t guarantee this – an OAuth JWT tells you about the user or client, but doesn’t prove that the network connection is from a trusted source. SPIRE bridges that gap by issuing **short-lived certificates** to each service, allowing them to establish **mutual TLS (mTLS)** connections. This gives us *cryptographic authenticity* of the calling service and encryption in transit. In a zero-trust network (where we assume the network is hostile), SPIRE makes sure that even within our backend, every call is authenticated and encrypted at the connection level [29](https://spiffe.io/docs/latest/spire-about/use-cases/#:~:text=However%2C%20for%20complex%20distributed%20applications,communication%20in%20potentially%20untrusted%20networks)[ 30 ](https://spiffe.io/docs/latest/spire-about/use-cases/#:~:text=distributed%20systems%20by%3A). Unlike network ACLs or IP whitelists (which can be spoofed) or shared secrets (which can leak), SPIRE certificates are tied to the actual runtime environment and rotate frequently![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.096.png) [31](https://spiffe.io/docs/latest/spire-about/use-cases/#:~:text=1,workloads%20via%20the%20Workload%20API)[ 32 ](https://spiffe.io/docs/latest/spire-about/use-cases/#:~:text=2.%20Delivering%20workload,workloads%20via%20the%20%2072).![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.097.png)

Key advantages of SPIFFE/SPIRE: - **No hard-coded credentials (“secret zero”):** Workloads automatically get credentials from the SPIRE agent when they start, based on attestation of the environment. For example, in Kubernetes, the SPIRE agent can use the pod’s service account or node identity to attest and issue a SPIFFE

ID for that pod [33](https://medium.com/@noah_h/cloud-native-identity-management-exploring-spiffe-spire-istio-89253cdb046a#:~:text=workloads%20is%20implemented%20via%20an,extensible%20plugin%20architecture)[ 34 ](https://medium.com/@noah_h/cloud-native-identity-management-exploring-spiffe-spire-istio-89253cdb046a#:~:text=Press%20enter%20or%20click%20to,view%20image%20in%20full%20size). The service code doesn’t need any password or API key baked in – it asks the local SPIRE agent for its certificate and key via a socket (Workload API)![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.098.png) [35 ](https://medium.com/@noah_h/cloud-native-identity-management-exploring-spiffe-spire-istio-89253cdb046a#:~:text=Document%20,509%20certificate%20or%20JWT). This solves the bootstrapping problem (no static secret distribution)![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.099.png) [36 ](https://medium.com/@noah_h/cloud-native-identity-management-exploring-spiffe-spire-istio-89253cdb046a#:~:text=The%20platform,applications%20to%20verify%20other%20workloads). - **Workload Authenticity:** Each service gets a **SPIFFE ID** of the form  spiffe://<trust-domain>/<service-name>/<etc> . For example, Service A in production might have  spiffe://saas.prod/serviceA . When Service A connects to Service B over mTLS, B sees that SPIFFE ID in the client cert and can enforce policies like “only services with ID spiffe://.../serviceA in domain

prod are allowed to call this endpoint.” This is much stronger than IP-based rules. It also thwarts attackers who may have stolen a JWT because without a valid SPIFFE certificate from our infrastructure, they can’t even establish a connection to our internal API endpoints. - **Prevents Lateral Movement:** If an attacker compromises one service, SPIRE ensures that service’s ID can’t be forged to access others. The attacker would have that service’s certificate (maybe), but not others. Also, certificates are short-lived (by default, often hours or less) and tied to specific attested environment characteristics (if the attacker moves the binary to another node, it won’t get the same cert). This significantly reduces the risk of an attacker pivoting within our microservices. The mutual TLS means the attacker can’t just sniff traffic or do man-in-the-middle between services – everything is encrypted and verified. - **Independence from user tokens:** Even if an OAuth token is stolen, if it’s presented from a source without a valid SPIFFE identity, we can reject it. For instance, imagine a user JWT leaked; an attacker tries to call Service B’s endpoint with that JWT. Service B will require that the transport cert is from our trusted SPIRE domain (say spiffe://saas.prod/\*). If the attacker isn’t in our cluster, they won’t have such a cert, so the connection fails. Thus, SPIRE adds a second factor of authentication for service calls (the service’s own identity).

**When to use SPIFFE vs OAuth:** It’s not either/or – they complement each other at different layers: - Use **OAuth/OIDC (JWTs)** at the application *request* layer – to know “*who* is the user or client making this request, and are they allowed to do this action?”. - Use **SPIFFE/mTLS** at the transport *connection* layer – to verify “*what* is the calling service and is it an allowed caller in the first place?”.

Consider clear trust boundaries: The primary trust boundary is between the outside world and our system. At that boundary (ingress), we rely on OAuth/OIDC for incoming user auth. *Inside* the system (behind the gateway or firewall), we establish a **mesh of trust** via SPIFFE. If a call stays entirely inside one trust domain (e.g. within one Kubernetes cluster or one VPC), we can rely on SPIFFE for service-to-service auth and not require JWTs for internal calls that don’t involve user context. If a call *crosses* a trust boundary (e.g. our cloud SaaS calling an on-prem instance or a third-party service), we then use OAuth (or some token mechanism) at that boundary.

**Coexistence in Architecture:** A secure architecture will use both: at the **edge**, we validate user JWTs; once the user is authenticated, the request enters the internal network where every hop is protected by mTLS with SPIFFE IDs. We might still pass the user JWT through internal calls (to know who the user is), but we don’t rely on it for the service identity. For example, the flow might be: 1. User calls our API Gateway with an access token. 2. Gateway verifies the JWT (user is authenticated) and also verifies its own caller (the user’s TLS, if mutual TLS with client cert is used in some cases, but usually just TLS). 3. Gateway then forwards the request to Service A. This internal call is over mTLS – Gateway presents its SPIFFE cert, Service A verifies it’s the trusted “gateway” ID. 4. Service A sees the user’s identity from the JWT (which Gateway might forward as is or in a header after validation). Now Service A needs to call Service B, including user context. It initiates an mTLS connection to Service B – its SPIFFE cert proves to Service B that *Service A* is calling. It also maybe sends along a user token (possibly an exchanged, down-scoped one as per section 1). Service B thus knows: the connection is from Service A (authenticated by SPIRE) **and** user X’s rights are delegated (from the JWT claims). 5. Service B responds, and so on.

In that flow, both layers are checked. We essentially achieve **defense in depth**: even if an attacker somehow got into the network, without a valid SPIFFE identity their calls will fail; and even if an internal service is tricked, the user JWT ensures they can only do what that user could. Each service will have an authorization policy that considers *both* the caller service and (if present) the user. For example, Service B might say “only accept requests from [Service A **acting as** an authenticated user with scope X] or [Service C with its own service account token]”.

**Trust Domains and Federation:** SPIFFE introduces the concept of a **Trust Domain**, which is essentially a namespace and trust root for identities. Our SaaS cloud environment might use a trust domain like![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.100.png)![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.101.png)

saas.prod  (so identities look like  spiffe://saas.prod/serviceA ). An on-prem deployment at a customer could be a separate trust domain  customer1.local . By default, identities in different trust domains are not trusted by each other. This is good for isolation – a cert from ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.102.png) customer1.local can’t call our cloud  saas.prod  services because the trust roots differ. If we ever need cross-domain calls (for

instance, our cloud support service reaching into an on-prem instance), we have two options: **federation** between SPIFFE domains, or fallback to OAuth. Federation is possible (SPIRE can be set up to trust another domain’s CA)[ 37 ](https://medium.com/@noah_h/cloud-native-identity-management-exploring-spiffe-spire-istio-89253cdb046a#:~:text=Having%20solved%20the%20identity%20management,various%20service%20mesh%20such%20as), but in practice it’s complex and requires coordination of root certificates. Often, it’s simpler to use an OAuth service account token for cross-domain communication. For example, our cloud might obtain an access token signed by the on-prem IDAM via a federation protocol, or just use a separate![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.103.png)

client credential that the on-prem instance accepts. In summary, *inside a trust domain, use SPIFFE; crossing trust domains, use OAuth (or carefully federated SPIFFE if needed).* This clear boundary ensures we don’t mix up trust configurations.

SPIRE takes care of rotating these certs frequently (say, every hour) to![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.108.png)

minimize the window if one were compromised. - **Istio or Service Mesh**: If we use a service mesh like Istio, it actually has SPIFFE built-in. Istio’s Citadel (now part of istiod) issues certificates with SPIFFE IDs to each![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.109.png)

sidecar  proxy.  So  Istio  mTLS  is  effectively  SPIFFE  under  the  hood  (identities  like  spiffe:// cluster.local/ns/default/sa/serviceA![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.110.png) ). Using Istio, we might offload mTLS to the sidecars and let

Istio enforce which services can talk to which (through AuthorizationPolicies that match SPIFFE IDs). This is a powerful approach if we already plan to use a mesh. - **DIY mTLS with SPIRE**: If not using a full mesh, each

service can fetch its cert from SPIRE and use it in its HTTP client/server. Many frameworks allow plugging in TLS certs. We would configure each service to trust the SPIRE CA for incoming certs and to present its own

cert on outgoing connections. The SPIRE Workload API has libraries for various languages, and there’s even an Envoy SDS integration to dynamically get certs. - **JWT-SVID**: SPIRE can also issue JWT-SVIDs (JWT-based identities)![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.111.png)[ 43](https://spiffe.io/docs/latest/spire-about/use-cases/#:~:text=While%20mTLS%20is%20useful%20for,of%20a%20corresponding%20public%20key)[ 44 ](https://spiffe.io/docs/latest/spire-about/use-cases/#:~:text=SPIRE%20supports%20a%20specific%20form,workloads%20in%20distributed%20systems%20by). These are JWTs signed by SPIRE that assert the service’s SPIFFE ID in the  sub . They can be used in cases where mTLS is not feasible – for example, if you have a message broker or a system that![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.112.png)

can’t do TLS but can attach a token. A service could grab a JWT-SVID from SPIRE and include it in an HTTP header for identification. However, since we already have OAuth JWTs for user/service permissions,

introducing SPIRE JWTs might complicate things. We likely stick to X.509 SVIDs (certs) for now. But it’s nice to know we have the option, e.g., if Service A needed to call Service B via an intermediary that terminates TLS

(like an L7 load balancer), Service A could sign a JWT-SVID and B could verify it using SPIRE’s JWKS. - **Workload Attestation**: We will configure attestors in SPIRE suitable for our environments. For Kubernetes,

the built-in k8s attestor can verify the pod’s service account token or use PSA tokens [45](https://medium.com/@noah_h/cloud-native-identity-management-exploring-spiffe-spire-istio-89253cdb046a#:~:text=For%20example%2C%20the%20,SVID)[ 46 ](https://medium.com/@noah_h/cloud-native-identity-management-exploring-spiffe-spire-istio-89253cdb046a#:~:text=K8s_psat,to%20tokens%20via%20volume%20projection). For VMs, we can use e.g. AWS IID attestor (which checks EC2 instance metadata to ensure it’s our EC2)![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.113.png) [33 ](https://medium.com/@noah_h/cloud-native-identity-management-exploring-spiffe-spire-istio-89253cdb046a#:~:text=workloads%20is%20implemented%20via%20an,extensible%20plugin%20architecture). This attestation is the root of trust – it means only our approved infrastructure can get identities. This is![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.114.png)

especially relevant for on-prem: on customer’s environment, we might run SPIRE with node attestation tied to their nodes (or even simpler, use a static attestor with config since it’s a closed environment).

**How SPIFFE and OAuth Work Together Securely:** - **Trust boundaries**: We treat the entire internal network as untrusted unless a call is authenticated via SPIFFE. Essentially, *every internal API is mTLS-only, accepting only our SPIRE-issued certs*. This prevents forgery and eavesdropping 4[7 ](https://spiffe.io/docs/latest/spire-about/use-cases/#:~:text=When%20sending%20messages%20between%20distributed,transit%20by%20a%20malicious%20party)4[8 . ](https://spiffe.io/docs/latest/spire-about/use-cases/#:~:text=particular%20manner%20and%20signed%20by,of%20a%20corresponding%20public%20key)At the same time, we don’t automatically trust a service just because it has a cert – we also enforce authZ based on JWTs. The SPIFFE ID tells us ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.115.png)*which service* is calling; the JWT tells us *who (user or client)* and *what for*. Both are needed for a complete picture. For instance, Service B might allow Service A to call a certain admin API *only if* Service A

presents both a valid SPIFFE cert and a valid JWT with an admin scope. If Service A’s credentials were compromised and an attacker tried to use them from a different machine, the SPIRE attestation would fail. If an attacker somehow got Service A’s cert (highly unlikely without root on the box) and tried to use it, it would soon expire and also likely be detected. - **Performance**: mTLS does add overhead (TLS handshakes, cert validation) but we mitigate that with HTTP/2 connection pooling, and the fact that inside a datacenter, the latency is low. SPIRE’s overhead is minimal – cert rotations are infrequent and handled asynchronously. The benefit of not having to hit an auth server on each request (thanks to JWTs) far outweighs the slight cost of TLS handshakes for new connections. We will also ensure our TLS config is modern (e.g. using TLS 1.3 which has faster handshakes). - **Security defaults**: All internal services *by default* require mTLS. If a new microservice is added, part of its onboarding is registering a SPIFFE ID and obtaining a cert before it can talk to others. We also use SPIRE to distribute the trust bundles (the CA root) to all services, so they know to trust only our CA. We log certificate issuance and could even use OPA (Open Policy Agent) with Envoy to do additional checks on SPIFFE IDs at runtime if needed [37](https://medium.com/@noah_h/cloud-native-identity-management-exploring-spiffe-spire-istio-89253cdb046a#:~:text=Having%20solved%20the%20identity%20management,various%20service%20mesh%20such%20as)[ 49 ](https://medium.com/@noah_h/cloud-native-identity-management-exploring-spiffe-spire-istio-89253cdb046a#:~:text=organizations%20spread%20across%20cloud,in%20production%20than%20explored%20below). - **Example real-world**: **Istio in PCI environment![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.116.png)**  –  many  financial  institutions  use  Istio  with  mTLS  (SPIFFE)  to  satisfy  strict  network authentication requirements, while still using OAuth tokens for user-level permissions. Also, **SPIRE** is used by Uber, Bloomberg, etc., to secure microservices. They often report that it eliminated entire classes of network spoofing attacks and simplified service authentication. We intend to achieve the same: if a developer spins up a new service, they don’t worry about sharing some secret with other services; as long as it’s registered in SPIRE, it automatically can authenticate to peers. And if a service should not talk to another, we can enforce that via SPIRE ID-based ACLs rather than brittle IP rules.

In summary, **SPIFFE/SPIRE gives our system a strong identity lattice for services**. It compliments the OAuth user-centric flows by ensuring that *services* themselves are verified and operating within allowed trust zones. This is especially important for on-prem deployments where the network may be less contained

- by shipping SPIRE with the product, even on a customer’s LAN the services will mutually authenticate and encrypt, mitigating risks of insider threats or compromised network devices. For the hosted cloud, SPIRE helps implement a zero-trust network (every call is verified), which is increasingly a requirement for high- security architectures (e.g. for SOC2, PCI, etc., to show segmentation and authentication of all internal traffic). 
4. **Service Accounts & Least Privilege**

Our platform must support not just human users but also **service accounts**  – these are non-human identities used by applications or integrations to programmatically call APIs. Designing service accounts with *least privilege*  in mind is critical, as a compromised service credential could otherwise become a “skeleton key.” Here’s how we approach service account management:

**First-Class Principals:** Service accounts are treated as full principals in the IDAM, much like users. Each service account has a unique identity (e.g., client ID or name), credentials for authentication, and is associated with permissions/roles. We do not piggy-back service auth on user tokens; instead, we issue dedicated tokens to services. This clarity allows separate tracking and revocation. For example, an external integration for “PayrollSync” will have its own client entry in IDAM (under tenant Acme Corp if it’s tenant- scoped) and get tokens that clearly identify it (e.g. ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.117.png) sub: payrollsync@acme-corp ). Internally, we might also use service accounts for microservices to call others (though with SPIRE we might not need OAuth for internal calls, but if we do token exchange, those tokens might use an internal service account identity as the actor).

**Credential Types:** A service account can authenticate in a few ways: - **Client Secret:** A shared secret (like a password) known to the service and IDAM. Used in simple scenarios (the service calls token endpoint with its ID and secret to get a JWT). Secrets are long random strings and can be rotated. This is standard OAuth2 client credentials flow. - **Client Certificate / JWKS:** A more secure option is to issue the service account its own key pair or certificate. For example, we could issue a signing key (or the service generates one and registers the public key). Then the service can request tokens by signing a JWT (JWT Bearer auth, RFC 7523) or by mTLS. We prefer this for high-security accounts to avoid symmetric secrets that might be leaked in config. On-prem, a service might even use SPIRE to get a certificate and then use that certificate to authenticate to IDAM (tying our internal trust with token issuance). - **API Key (as a variant):** For some external integrations, we might issue an API key that the client uses directly to call our API (as a header). Under the hood, we validate it and treat it as a service account token. This is mostly for simpler integration scenarios or when OAuth is not feasible for the integrator. However, we will encourage full OAuth flows for any serious integration.

**Least Privilege Scoping:** Every service account is created with a specific scope of access. By default, *no* service account has global or admin access (avoid the “god-mode” token scenario). We achieve least privilege through: - **Tenant Association:** If an external integration is meant for a single tenant, we bind the service account to that tenant. Its tokens will carry the ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.118.png) tenant: X claim just like a user’s would. The IDAM

will ensure it cannot obtain a token for another tenant. For instance, if “PayrollSync” is set up by Acme Corp’s admin in their tenant space, that service account’s tokens always have ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.119.png) tenant = acme-corp . Even if the integration tried to ask for data from another tenant by ID, our services would reject it because the

token’s tenant doesn’t match. - **Audience & API Scope Restrictions:** When registering a service account, the admin or system must specify what API scopes it needs. If PayrollSync only needs to read and update![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.120.png)

payroll-related  data  via  a  certain  API,  we  give  it  only  those  scopes  (e.g.,  payroll.read , ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.121.png)

payroll.update on the Payroll API). The service account will not be able to get a token with any other scope – IDAM’s token endpoint will enforce an allowlist. Moreover, tokens issued to that client will have![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.122.png)

aud fixed to the Payroll API. If the integration tries to use the token on another API, it won’t work. This mechanism is akin to Auth0’s “RBAC for APIs” where you assign permissions to a client and its tokens reflect only those permissions. - **No Implicit Superuser:** We avoid patterns where a service account has a blanket “admin” role across the platform. Instead, if we have internal maintenance tasks that truly need broad access, we handle them by orchestrating calls under controlled conditions (for example, an internal “maintenance tool” might iterate through tenants, but at each step it obtains a token scoped to that tenant’s data only). In other words, even our internal automation should abide by tenant boundaries whenever possible, rather than holding a universal credential.

**Preventing God-Mode Tokens:**  A “god-mode” token would be one that, if acquired, grants access to everything. We prevent this in multiple ways: - **Split Responsibilities:**  Different services have different service accounts. There isn’t one single secret that unlocks all APIs. For internal microservices, each has its own identity (which SPIRE enforces) – they communicate under mutual authentication; for cross-service OAuth calls (if any), Service A’s token for Service B is limited as described. For third-party integrations, each integration has its own account with limited scope. - **No Wildcard Scopes:** We do not issue tokens with wildcard scopes like ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.123.png) \*.\* (all access). Even admin services have to enumerate what they need. If at some point we have a central admin integration (say a command-line tool for our ops team), we might give it a broad set but still enumerated (like a role that has every permission explicitly listed, rather than an implicit all). - **Expiration & Monitoring:** Service account tokens also expire quickly. Long-lived tokens (like an API key) are only used if absolutely needed, and even then, they can be rotated and are tied to an account we can disable. We monitor usage of service accounts – e.g., if a usually quiet service account suddenly starts calling lots of APIs, that’s a red flag. This monitoring, while not directly an architecture component, is part of maintaining least privilege in practice.

**Delegation vs Impersonation for Services:** Sometimes a service account needs to perform an action *on behalf of a user*. A prime example is a backend batch process that impersonates users to perform some deferred work (maybe send notification as if from that user). To handle this properly, we again use the ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.124.png)**OAuth Token Exchange** with the  act claim. A service can be granted an ability to impersonate within certain bounds. We have two patterns: - **Delegation (Recommended):** The service account obtains a token that represents the user but also itself as actor (![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.125.png) act ). This way, when it calls downstream, the services see user = Alice, actor = ServiceX. They can log both. If needed, certain operations can be restricted to only real- user contexts (e.g., “only a real user token with no actor can delete an account”). We configure IDAM to only allow specific service accounts to perform token exchange, and only for certain scopes. For instance, the “ReportingService” might be allowed to get a token for user data ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.126.png)*but only read-only scopes* and with an  act indicating it’s the reporter. This is done via IDAM policy (could be using the ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.127.png) may\_act claim in the subject token to restrict who can use it [11 ](https://docs.pingidentity.com/pingam//7.2/oauth2-guide/oauth2-token-exchange.html#:~:text=the%20party%20or%20parties%20allowed,see%20About%20the%20may_act%20claim)). - **Impersonation (Cautious use):**  In rare cases, a service might completely impersonate a user (no act claim). This could be useful if, say, we need to trick a third-party library or system that only accepts tokens directly from users. Keycloak’s “impersonation” feature issues a token as if the admin were the user. We will use this only internally and with short lifespan. The logs in IDAM itself will note it was issued by admin X on behalf of user Y. Downstream services won’t know it’s impersonation, so we limit what such tokens can do (e.g., maybe not allow critical transactions without an additional check). The general rule is to ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.128.png)**prefer delegation** (maintain the chain of actors) for transparency.

**Just-In-Time (JIT) Privilege Elevation:** Service accounts may sometimes need a privilege only at a specific moment. Instead of always having that permission, they can request it on the fly. For example, consider an internal “MigrationService” that normally just reads data, but occasionally needs to write to all tenants’ data during a migration window. Rather than giving it permanent write access (god mode), we can implement a flow where it requests an *elevated token*  at runtime from IDAM with a special grant (which could even require a human approval or a feature flag). IDAM then issues a token with the elevated scope valid for a very short time (say 5 minutes) which the service uses and then discards. This JIT approach ensures that even if MigrationService were compromised at a random time, it wouldn’t have write access unless the attacker also triggered the special elevation process (which we could guard). This pattern is drawn from privileged access management principles in high security contexts (just like how human admins might get an elevated role only after just-in-time approval).

**Cross-Tenant Safeguards:** One of the biggest risks with service accounts is that an internal service that has access to all tenants (by design) could be abused to leak data across tenants. We mitigate this heavily: - **Internal services with multi-tenant reach** (like an analytics aggregator) should ideally *not operate under a single blanket token*. Instead, if Service X must iterate over tenants, it should programmatically obtain a token for each tenant in turn (possibly using an internal admin user context or a tenant-scoped client credential). For instance, an “Email Blaster” service that sends announcement emails to all customers could loop through the tenant list, and for each, acquire a token that only allows email sending for that tenant. This ensures that if midway someone intercepts the token, it only works for that tenant. - If obtaining per- tenant tokens is too slow, another approach is to embed the tenant id in each request that the service makes internally and have the receiving service validate that tenant against an allow-list for that service’s principal. E.g., Service X calls Service Y with a special service token that has a claim ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.129.png) tenant\_scope: ["tenantA","tenantB",...]![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.130.png) . We strongly limit which tenants can appear there. Perhaps in a SaaS scenario, internal services run in a “platform” tenant context that essentially has access to all, but then every

such usage is gated by code checks (e.g., require an explicit tenant parameter and audit). - **Acting as no- tenant principal:** Alternatively, internal cross-tenant services might bypass normal tenant scoping but only through designated internal APIs. For example, Service B might have an internal endpoint ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.131.png) /all-tenants/ recomputeCache![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.132.png) that only accepts calls from the Platform’s SPIFFE ID *and* a special internal token. This is not exposed to normal tenants. In effect, we separate the interface used for multi-tenant operations from the one used per-tenant, reducing risk of accidentally calling the wrong one. - **Policy and Monitoring:** IDAM can enforce that a given service account can’t get tokens for a tenant it’s not associated with. For user-impersonation flows, it can ensure Service X can only impersonate users of tenants X, Y, Z. If Service X tries to ask for a token for tenant Z’ (not allowed), IDAM denies it. These rules would be configured when setting up the service account (e.g., “allowed\_tenants”: [“acme-corp”] for PayrollSync, or [“*”] for an internal platform service that is explicitly allowed globally). - Audit trails:*\* Every use of a service account is logged with which tenant context was accessed. We maintain logs like “ServiceAccount=BillingReconciler used TokenID 123 to read invoices for Tenant=AcmeCorp at time T”. This way, if anything fishy happens (like a service account accessing a tenant it normally doesn’t), we catch it.

**Use Cases:** - *Internal Worker Example:* Consider an **Invoice PDF Generator** service that generates invoices for all tenants. We give it a service account that has permission “invoice.read” and “invoice.generate\_pdf”. It does not have “invoice.create” or other powerful actions. When it runs, it queries each tenant’s invoices and produces PDFs. If it’s compromised, worst case it can read invoice data (which is sensitive, but it can’t modify or delete anything). We might even split reading vs writing – maybe it calls an internal API in each tenant context to fetch data (with a token per tenant as above). The service account cannot arbitrarily call user-facing endpoints; it uses special internal endpoints (or an admin API) that still validate its scopes. - *External Webhook Integration:* Suppose a customer (tenant) wants to integrate their Slack app to receive alerts from our SaaS. We’d have them create a service account (or we pre-create integration types). The Slack integration service account might have permission only to read certain data and send it out, or maybe to receive events from Slack and call a specific endpoint. We ensure that this service account is linked to that tenant and can’t accidentally access another tenant’s info. Technically, this could be done with an OAuth client in our IDAM: the customer’s admin goes to our UI, creates a new “Slack Integration” client, gets a client ID/secret. In Slack’s app configuration, they store those and use them to get an access token from our IDAM via client credentials. That token then might allow calling only ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.133.png) /tenant/{acme}/alerts endpoint on our API. If the Slack app token were stolen, the damage is limited to reading or sending alerts for Acme tenant only. We also encourage customers to rotate secrets regularly and can integrate with their systems (for example, they could use certificate-based auth instead of static secret). - *Platform vs Tenant Service Accounts:* We conceptually separate “platform” service accounts (those used by our SaaS company’s internal services that might span tenants) and “tenant” service accounts (created by tenants for their own integrations). Platform service accounts are configured by us and have very limited use cases (with all the checks described). Tenant service accounts are under each tenant admin’s control – they decide to create one, assign it certain roles (which map to scopes), and they see an audit of its usage. The IDAM provides an interface for this in the SaaS UI. On-prem, the customer’s admin would do similarly via the bundled IDAM UI or CLI.

To sum up, **service accounts are isolated, scoped, and controlled identities**. By treating them with the same rigor as human users (unique IDs, least-privilege permissions, 2FA where applicable, monitoring), we avoid them being backdoors. This design draws parallels with cloud IAM practices: e.g., in AWS you don’t log in with root credentials for automation; you create an IAM role with specific policies. Similarly, our IDAM issues tokens with specific policies for each service principal. There’s no all-powerful master token traveling around. Even the central IDAM itself, when calling other services (if it does, e.g., provisioning something), would use a scoped token or be recognized by SPIFFE rather than having carte blanche.

5. **Central IDAM vs Service Autonomy**

In  a  multi-tenant  SaaS  with  many  microservices,  a  key  challenge  is  managing  authorization  data (permissions, roles) in a scalable way. We need a **central Identity & Access Management (IDAM)** system that provides a single point of truth for user identities, role assignments, and so on. However, we also need to give **services autonomy**  to define the fine-grained permissions that actually matter to them. The philosophy is: **services define *permissions*, IDAM defines *roles* and orchestrates them across services, and tenants control *assignments*.**

**Services Define Permissions (Not Central IDAM):**  Each microservice knows its own domain best and should declare what actions can be performed. For example, a Billing service might define permissions like![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.134.png)![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.135.png)

invoices.create ,  invoices.pay ,  invoices.view ,  etc.  The  Inventory  service  might  define

stock.update ,  stock.view  etc. These are the atomic rights within that service. The central IDAM doesn’t hard-code these – instead, there’s an API or schema where services register their permission names and descriptions. This registration can happen at deployment time or dynamically. For instance, when the Billing service starts up (or through CI/CD), it calls an IDAM endpoint to publish “here are my permissions and what they mean.” The IDAM stores these in a repository of all permissions available in the system, tagged by service. 

By doing this: - We ensure **consistency** – there’s a single canonical list of actions per service, which can be presented in UIs or used in tokens (scopes). - We allow **extensibility** – if a new microservice is added or an existing one evolves, it can introduce new permissions without requiring a monolithic auth system update. The service team can add a permission in code, expose it via the registration, and start enforcing it. IDAM just stores and propagates it. - We keep enforcement logic **local**  – the microservice still enforces the permission at runtime (e.g., check “does token have invoices.pay?” before executing payment). The IDAM’s role is to distribute knowledge of who should have that permission via roles or grants.

This approach mirrors AWS IAM in spirit: AWS services each define their own set of “actions” (e.g.,![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.136.png)![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.137.png)

s3:PutObject ,  ec2:DescribeInstances ). AWS IAM (the central system) doesn’t have an internal list of all possible actions by itself; it’s driven by the services. Our IDAM will similarly act as a policy store but rely on services to provide the action vocabulary [50 ](https://aws.amazon.com/blogs/apn/tenant-switching-and-custom-permissions-in-a-multi-tenant-serverless-application/#:~:text=In%20summary%2C%20an%20IAM%20policy,principal%20tag%20variables).![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.138.png)

**Role Templates Scaffolded by Services:** While individual permissions are great for granular control, in practice admins prefer to assign roles (groups of permissions) especially if there are many permissions. We enable microservices to provide **role templates** – suggested collections of permissions that make sense in that service’s context. For example, the Billing service might suggest a role called “BillingManager” that includes![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.139.png)![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.140.png)  invoices.view  and  invoices.pay ,  and  another  role  “BillingClerk”  that  only  has

invoices.view . It might also define a role “BillingAdmin” with all its perms. These are essentially *blueprints*. The service registers these templates with IDAM, referencing the permissions it defined.

The central IDAM can then use these building blocks to create actual roles in a tenant context: - If a role template is **tenant-independent** (like it only concerns one service), the IDAM can instantiate it per tenant. E.g., every tenant gets a “BillingManager” role available which they can assign to their users as they see fit.

It has the permissions that the Billing service specified. If the service updates the template (e.g., adds a new permission to BillingManager), the IDAM can propagate that change to all tenants’ roles (with admin approval if needed). - If roles span multiple services, it might be up to the tenant admin or central admin to compose them. For example, we might have a cross-service need like “SupportAgent” role that includes read access to CRM service, Ticketing service, and maybe limited write on one. The central IDAM can allow creation of such composite roles by combining permissions from various services (or combining service- specific role templates).

**Tenants Control Role Composition and Assignment:** Each tenant is effectively a silo when it comes to *which users have which permissions*. The IDAM provides an interface where a tenant admin can manage roles in their organization: - They can use out-of-the-box role templates provided by services. For instance, they see “BillingManager (template)” and can adopt it as is – meaning all the permissions that template entails. - They can customize or create new roles. Perhaps Acme Corp wants a role “FinanceExecutive” which includes both the BillingManager perms and some from Accounting service. They can create that in IDAM by picking permissions (the IDAM UI would list all available perms, grouped by service). This is analogous to AWS IAM where you can pick actions from different services to build a policy, or Keycloak where you can make composite roles from roles of multiple clients. - They assign roles to users or service accounts in their tenant. The IDAM then issues tokens or assertions that reflect those assignments (as discussed in JWT section, typically by giving the union of the permissions in the token’s scope or by an opaque link to fetch them).

One key design: **services do not make final authorization decisions based on “roles”** – they base it on permissions. So if Tenant admin created a custom role “FinanceExecutive” that includes ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.141.png) invoices.pay , when a user with that role calls the Billing API, the JWT will have ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.142.png) invoices.pay (either directly as a scope

or because IDAM sees the role includes that permission and includes it in the token). Billing service doesn’t care what the role name is, it just sees the permission. This decouples service logic from the role naming and structuring that each tenant might do. It’s a security measure too: it ensures no “hidden” permissions – services only consider the concrete actions.

**IDAM as Mediator of Role/Permission Registration:** We implement an **API surface** in IDAM for services to register their auth model pieces: - ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.143.png) RegisterPermissions(service, permissions[]) – probably done at service startup or via CI deploy. It could include metadata like descriptions and whether each perm is![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.144.png)

considered safe for read-only roles, etc. -  RegisterRoleTemplates(service, roles[]) – where each role entry has a name, description, and list of permission keys (which were registered earlier). - Possibly a![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.145.png)

RegisterResourceTypes if we get into more fine-grained resource-based permissions (like “permission to view *own* record vs any” – that could be a dimension, but we may handle that via policies in service code rather than IDAM knowing about resources).

The IDAM, upon receiving these, will store them (likely keyed by service and maybe a version). If a service registers a permission that already exists globally under that name, IDAM could either prevent duplicates or namespace them by service internally. Namespacing is helpful to avoid collision – e.g. two services might both![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.146.png)  have  a  view  permission  that  should  be  distinct,  so  likely  we’ll  namespace  as![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.147.png)

billing.invoices.view vs  inventory.items.view . 

**Service Versioning and Compatibility:** If a service deprecates a permission or changes meaning, it should update the registration. IDAM might keep old ones marked deprecated (so that existing roles don’t break). This requires some governance – likely our engineering process will handle that (e.g., communicate that “permission X is now replaced by Y for new features” and we migrate roles accordingly).

**Real-World Parallels:** - *AWS IAM:* As mentioned, AWS IAM serves as a central broker of permissions defined by each service. Our IDAM’s model is similar: microservices publish their actions, IDAM lets admins (the tenants or us as root) assemble those into roles/policies. The concept of **“managed policies”**  in AWS (predefined sets of actions for convenience) is analogous to our role templates from services. For example, AWS might have an AWS-managed policy “AmazonS3ReadOnlyAccess” that is a collection of S3 actions – S3 service team defined it. In our case, Billing service defines “BillingReadOnly” template etc. - *Keycloak Composite Roles:* Keycloak supports roles at two levels: realm roles and client roles. Each microservice in our case could be represented as a Keycloak “client” with its own roles (permissions). Keycloak allows you to create composite roles that aggregate other roles. So one could make a composite that includes roles from multiple clients (services). This is exactly how a cross-service role would be built. For instance, you could have![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.148.png)  a  realm  role  “SupportAgent”  that  composites  CRM\_ReadOnly  (role  on  CRM  client)  +![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.149.png)

Tickets\_ReadWrite (role on Ticketing client). Keycloak will then put all those in the token if configured, or use its policy engine to evaluate. Our system is conceptually similar but purpose-built for multi-tenant: each tenant corresponds to an isolation of role assignments, and our IDAM would essentially implement a multi-tenant aware composite role mechanism. - *Supabase and Postgres RLS:*  Supabase doesn’t have a traditional role/permission UI, but it leverages Postgres roles and row-level security (RLS). The interesting parallel is that each component (the database schemas in that case) defines what it means by a role or permission (in SQL policies), and the JWT just passes along a role claim. If one wanted to have multiple roles in Supabase JWT, they’d have to encode them in an array claim or do lookups in the DB. In our case, rather than rely on a single role text, we explicitly manage multiple permissions. However, the enforcement at the service is akin to RLS policies: the service checks the claim against some logic. - *Policy Engines (OPA, etc.):* As we scale, some authorization might become too complex for simple role-perm matching (like context-aware rules: “Managers can approve expenses up to $1000” etc.). Our architecture anticipates that by keeping IDAM focused on identity, tenant, and static permissions, while allowing **services to implement finer policies** on top. For example, our RBAC service (from RERP’s mention) might evolve to an **ABAC** (attribute- based) or **Rego policies** stored centrally but evaluated in services or a sidecar (could integrate OPA with SPIFFE as shown in some guides [49 ](https://medium.com/@noah_h/cloud-native-identity-management-exploring-spiffe-spire-istio-89253cdb046a#:~:text=organizations%20spread%20across%20cloud,in%20production%20than%20explored%20below)). However, that’s an extension; core roles/permissions still form the base.![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.150.png)

**IDAM & Service Coordination:** The central IDAM also acts as a **directory of services** in terms of auth capabilities: - It can expose an endpoint “what can principal X do?” which compiles across services. This would be useful for an admin UI where, say, you want to show all permissions a user has across the system. IDAM can look at roles assigned to the user and explode them into permissions from all services (since it has the mapping). For example, if Jane has role “BillingManager” in tenant Acme, IDAM knows that means she has ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.151.png)![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.152.png) invoices.view and  invoices.pay on Billing service. It could return that list. - It also can enforce consistency like ensuring no duplicate permission keys, and possibly link related permissions. It might know some hierarchy (if defined). For instance, maybe a service defines a hierarchy (like in RBAC README they mentioned hierarchical roles [51 ](https://github.com/microscaler/rerp/blob/d9800433f9aab3298de2a86637e5b716e6d94463/openapi/auth/rbac/README.md#L43-L50) and fine-grained resource/action/field levels [52 ](https://github.com/microscaler/rerp/blob/d9800433f9aab3298de2a86637e5b716e6d94463/openapi/auth/rbac/README.md#L44-L48) – IDAM could store metadata but enforcement is service’s job). - We might also enable ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.153.png)![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.154.png)**permission discovery**: tenants can query IDAM “what permissions does service X have?” and get a list to decide custom roles. This is analogous to AWS’s IAM Policy Generator UI where you pick a service and it lists all actions for you to select.

**Multi-Tenancy in IDAM:**  Our central IDAM is itself multi-tenant aware: each tenant’s admin can only manage their users and roles. The IDAM ensures isolation – a tenant admin cannot create roles that include permissions for a service they don’t subscribe to, perhaps. (If some tenants don’t use a module, we might hide those perms.) Also, if a tenant is deleted or offboarded, IDAM cleans up assignments.

**Example:** Let’s walk through adding a new microservice and using this system: - We create a new service “AnalyticsService” that provides reporting. The developers decide on two permissions: ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.155.png) reports.view and

reports.manage  (to create custom reports). They also suggest a template role “Analyst” = just

reports.view , and “AnalyticsAdmin” = both perms. - During deployment, AnalyticsService calls IDAM’s RegisterPermissions  with  {service:  "analytics",  permissions:  [{key:"reports.view", desc:"View  analytics  reports"},  {key:"reports.manage",  desc:"Create/Edit reports"}]} .  It  then  calls  RegisterRoleTemplates  with {service:"analytics",  roles: [{name:"Analyst", perms:["reports.view"]}, {name:"AnalyticsAdmin", perms: ["reports.view","reports.manage"]}]} .  -  IDAM  stores  these.  Now  IDAM’s  admin  UI  can  list “Analytics Service” with those roles. - For each existing tenant, the IDAM could either automatically add those role templates to their catalog (maybe marked as new and not assigned to anyone yet), or tenant admins will see them as available to assign. Perhaps we send a notification “A new feature Analytics is available with these roles”. - Tenant ACME’s admin decides to assign “Analyst” role to Bob. In IDAM, under Acme tenant, Bob gets role = Analyst@Analytics. IDAM knows that means Bob has![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.156.png)  reports.view permission. - When Bob logs in or his token is refreshed, IDAM includes  reports.view in his JWT (since he has that permission via role). - AnalyticsService receives JWT, sees  reports.view , allows Bob to fetch reports. If Bob tries an action requiring ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.157.png) reports.manage , he’ll get a 403 because that perm isn’t in his token. - If ACME admin wanted a custom combo, they could create a role “AnalyticsPowerUser” in their tenant that includes ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.158.png) reports.view and some other service’s perm – they can do that because IDAM lists all perms (assuming they have rights to that other service’s perms). IDAM stores this composite for ACME only, and Bob (or whoever) can be assigned to it. - If later AnalyticsService adds a new permission![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.159.png)

reports.delete , and updates the “AnalyticsAdmin” template to include it, IDAM gets updated. Possibly IDAM could auto-update tenants that were using AnalyticsAdmin role template to now include the new perm (maybe with an opt-in from admin if it’s sensitive). This way, service-driven changes propagate.

**Security-First Defaults in IDAM:** - By default, new permissions could be **off** for all users until assigned. We won’t auto-give everyone a powerful new permission. It has to be added to a role and granted by an admin.

- Role templates from services are just suggestions; tenant admin can choose to ignore or modify them. E.g., they might copy “BillingManager” template into a custom role and then remove one permission if they don’t like it. We allow that flexibility because each tenant might have slightly different least-privilege needs.
- If a service is critical (financial data), we might require certain roles to be split (e.g., no single role template includes both view and pay; maybe they intentionally separate for Segregation of Duties – our system should allow that but not enforce by default). - We ensure all permission changes are audited: if someone in IDAM changes which perms are in a role, we log that action (who, when, before/after). This is necessary for compliance (SOX, etc.) to track authorization changes.

**Comparative Note:** Another approach is to put all enforcement in one place (like a giant policy service). We deliberately avoid a single monolithic policy engine for everything because each microservice might have domain-specific rules. Instead, we give services the building blocks (permissions) and have IDAM coordinate the assignment. It’s a balance of central control and distributed enforcement: - The central IDAM is the **policy decision point**  for “who has what high-level access” (via roles/permissions issuance). - Each microservice is the **policy enforcement point**  for requests (checking tokens for required perms, and possibly other context like “is this user the owner of the record?” which the service knows best).

This way, if one service goes down, others still enforce their own auth with the cached token info; if IDAM goes down, existing tokens still work and services enforce, though new logins might not happen until it’s up. We avoid single points of failure and allow graceful degradation (e.g., if IDAM temporarily can’t introspect, tokens still carry enough info to continue operations for a short time).

**Parallels in Products:** - Some IdAM products like **Auth0** provide a feature called “Organizations” where each org (tenant) can have its own roles and role<->permission mappings. Under the hood, Auth0’s “permissions” are often just API scopes, and roles group them. That is conceptually what we’re doing, just that our “permissions” are numerous and come from microservices. - **AWS Organizations & IAM** – AWS allows an enterprise to define custom permission sets and use them across accounts. In our context, think of each tenant as an account, and our central IDAM as the organization that ensures consistency of permission definitions across all tenants. - **Supabase & RLS** – On a different note, Supabase’s approach could be seen as service autonomy to the extreme: each service (or table) defines who can do what in SQL, and the central auth just gives an ID and role. We take a more structured approach but still allow each service to say “these are the operations I care about.”

In conclusion, our **central IDAM vs service autonomy** model ensures **security and agility**: - Security, because nothing falls through the cracks – every permission is accounted for in IDAM’s registry and can be audited; and tenants can’t accidentally assign something that doesn’t exist or miss something that does. - Agility, because services can introduce new capabilities without waiting on a central authority to catch up – as long as they register it, it’s available. It’s very much like a plugin model for authZ. - And importantly, **tenants have full control** over how permissions are combined into roles for their users, which respects that each customer may have their own organizational policy. We as the SaaS provider supply sane defaults (through service-provided templates akin to best-practice roles), but the tenant can tweak to their needs. This approach is a **security-first default with flexibility**: out of the box, least-privilege roles are defined (so a tenant admin can just use them and not accidentally over-provision), but if they need, they can tighten or adjust the roles.

By adopting this model, we meet compliance needs (clear separation of duties, audit logs of who has access to what) and we make the system manageable at scale (imagine 100 microservices each with 10 permissions – we’d have 1000 permission items; our IDAM helps organize and present these logically to admins, rather than overwhelming them). ![](b01b1447-19d5-4ae0-bcff-afb64456a380_00000.160.png)

**Conclusion:** This comprehensive authentication and authorization architecture uses **OAuth 2.1 / OIDC** for end-user and service auth, **JWTs** as a flexible token format with tenant-aware claims, **SPIFFE/SPIRE** for strong service identity and mTLS protection internally, strict **service account scoping** for any programmatic access, and a **centralized IDAM** that orchestrates roles/permissions while empowering each microservice to enforce security at the point of use. It is designed with security-first principles (zero trust networking, least privilege by default, short-lived credentials, auditable actions) which align with PCI and financial industry practices. Moreover, it remains practical: leveraging standards and real implementations (like Keycloak, Istio, etc.) where possible instead of reinventing the wheel. The result is a multi-tenant SaaS identity architecture that can be deployed both in the cloud and on-premise with equal rigor, giving every stakeholder (the SaaS provider, the tenant admins, and even external integrators) confidence that authentication and authorization are robustly handled across the board.
